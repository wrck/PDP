# ADR-0001：采用模块化单体

- 状态：已接受
- 日期：2026-07-17
- 追溯：宪章原则 IV、plan.md“架构与模块分解”

## 背景

PDP P1 需要同时交付工作空间、领域配置、项目交付、审批、工作流、迁移和运维能力。首期团队与容量证据不足以支持微服务带来的独立部署、分布式事务和跨服务一致性成本，但必须保留后续按容量或组织边界拆分的路径。

## 决策

后端采用单一可部署应用和强模块边界。业务模块只能通过公开应用服务、只读查询端口或领域事件协作；禁止跨模块访问内部表、Mapper 和实现类。核心事务在单库本地事务内完成，非关键协同通过事务事件异步执行。Spring Modulith 与 ArchUnit 作为持续门禁。

## 后果

- 优点：事务简单、部署和排障成本低、可快速形成纵向闭环。
- 代价：必须严格治理模块依赖，后台作业与在线请求共享制品资源。
- 数据影响：模块拥有自身逻辑数据，公共变更集不代表允许跨模块直接查询。

## 替代方案

- 微服务：当前缺少容量、隔离和组织证据，不采用。
- 无边界分层单体：容易形成共享表和循环依赖，不采用。

## 验证与拆分触发

模块边界测试、循环依赖检查和端到端试点必须通过。当单模块容量、故障隔离、合规或团队所有权持续无法由模块化单体满足时，提交新 ADR，先稳定契约和数据主权，再拆分部署。
