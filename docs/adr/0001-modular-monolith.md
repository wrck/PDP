# ADR-0001：PDP 采用模块化单体架构

## 1. 状态与日期

- **状态**：Accepted
- **日期**：2026-07-17
- **决策人**：架构负责人（主笔）、产品负责人、领域负责人、运维/SRE 负责人、安全与合规负责人
- **适用范围**：PDP（Project Delivery Platform）P1 实现基线
- **关联文档**：`specs/002-pdp-product/spec.md`、`specs/002-pdp-product/plan.md`、`.specify/memory/constitution.md`、`specs/002-pdp-product/technology-comparison.md`

## 2. 背景

PDP 是面向企业项目交付的统一管理与扩展平台，P1 必须在单一可上线版本中交付完整业务闭环，覆盖 14 个业务与运行模块（含 `workflow`）、1 个公共持久化基础模块、1 个 MySQL 适配器、1 个标准实施领域包、1 条端到端试点以及 1 套 MySQL 历史数据迁移与切换方案。

依据 `plan.md` 的“架构与模块分解”，P1 业务模块矩阵如下：

| 模块 | P1 职责 | 主要追溯 |
|---|---|---|
| `identity` | 企业身份映射、会话、用户停用、授权版本和撤销时限 | FR-063～FR-069、FR-164 |
| `workspace` | 工作空间、组织、成员、角色、跨空间授权 | FR-003～FR-006、FR-121～FR-124 |
| `domainconfig` | 领域包、对象/字段/页面/状态/规则、版本、发布和迁移 | FR-007～FR-020、FR-130～FR-134 |
| `template` | 项目模板及批准版本 | FR-021～FR-022 |
| `project` | 项目、阶段、父子项目、顶层生命周期 | FR-023～FR-028、FR-117～FR-120 |
| `planning` | 任务、检查项、里程碑、依赖、基线、进度 | FR-029～FR-037、FR-125～FR-129 |
| `deliverable` | 交付件、版本、发布、签核和归档清单 | FR-038～FR-042 |
| `approval` | 统一审批定义、实例、节点、委托与回写 | FR-043～FR-046 |
| `workflow` | 平台级 Flowable BPMN 基础能力 | FR-009、FR-017、FR-043～FR-046、FR-174 |
| `governance` | 风险、问题、变更、审计、数据保留 | FR-047～FR-052、FR-069～FR-071 |
| `experience` | 列表、看板、日历、时间线、保存视图、基础搜索与站内通知 | FR-053～FR-060 |
| `standarddelivery` | 标准实施领域包及从创建到归档的闭环 | FR-094、FR-135～FR-138 |
| `integration` | OIDC、对象存储、内部事务事件、幂等重试和健康度基础 | FR-063、FR-066～FR-069 技术支撑 |
| `operations` | 可用性、性能、后台作业、备份恢复、降级、SLI 和运行观测 | FR-102～FR-111、FR-165 |
| `datamigration` | MySQL 盘点、映射、全量/增量迁移、隔离、核对、切换与退役 | FR-139～FR-150 |
| `persistence` | MySQL 能力检测、公共持久化契约、方言与变更集及适配边界 | 宪章原则 V、FR-139～FR-150、FR-171～FR-172 |

P1 同时承担以下非功能约束：

- **并发与性能**：1000 名并发活跃用户；95% 常用页面在 2 秒内呈现可用结果；百万级记录下 95% 常用搜索和报表在 3 秒内返回或先呈现渐进结果。
- **水平扩展**：应用层必须支持多副本水平扩展，核心业务可用性不低于 99.95%，RTO 不超过 30 分钟、RPO 不超过 5 分钟。
- **单写主权**：任一生产部署同一时刻只能有一个业务写入主库；核心事务不得依赖搜索、报表、AI 或非关键外部系统。

根据宪章原则 IV（稳定内核、受治理扩展与简单演进），架构 **MUST 从满足当前质量属性的最简单方案开始**；新增中间件、服务拆分、跨库协同或自研框架 **MUST** 由容量、隔离、合规或组织边界证据驱动，并通过 ADR 记录背景、选项、后果和退出条件。`plan.md` 的“复杂度与例外跟踪”也明确指出：模块化单体、每部署单一写入主库、P1 单一 MySQL 适配器、数据库无关端口和标准 HTTP/事件契约是满足当前范围的最小可行复杂度；消息代理、独立搜索集群和微服务拆分均延后到容量数据或组织边界证明必要时。

因此，P1 启动前必须确定一个满足上述质量属性、保留未来演进路径、且不引入未经验证复杂度的架构基线。

## 3. 决策

采用 **模块化单体（Modular Monolith）**：PDP 后端构建为单一可部署的 Spring Boot 4.1 应用，在制品内部以 Spring Modulith 2.1 建立强模块边界。

具体约束：

1. **单一可部署制品**：PDP 应用作为单个 Spring Boot 4.1 制品构建与发布，承载同步命令、查询、权限决策和核心事务，并保持无状态以支持多副本水平扩展。
2. **强模块边界**：模块之间 **只能** 通过以下三种受控通道协作：
   - 其他模块公开的 **应用服务**（Application Service）；
   - 其他模块公开的 **只读查询端口**（Read-only Query Port）；
   - **领域事件**（Domain Event，基于 Spring Modulith JDBC 事件仓储）。
   禁止跨模块直接访问内部表、Mapper、`BaseMapper`/`IService`/`QueryWrapper` 等框架类型或实现类。
3. **后台任务执行器**：与 PDP 应用共享同一制品与同一模块集合，但以独立运行配置（`profile`）启动，处理 Flowable 异步作业、导入导出、索引、通知、迁移和集成重试，不单独发布制品。
4. **Flowable 流程引擎**：作为 `workflow` 模块嵌入 PDP 应用，仅启用 Process Engine，使用独立 `workflowEngine` schema/账号/HikariCP 池；不直接暴露 REST API，不保存权威业务结论。
5. **后端模块清单**（P1）：`shared-kernel`、`identity`、`workspace`、`domainconfig`、`template`、`project`、`planning`、`deliverable`、`approval`、`workflow`、`governance`、`experience`、`standarddelivery`、`integration`、`operations`、`datamigration`、`public-persistence`、`persistence-mysql`。其中 `public-persistence` 维护公共持久化契约与端口，`persistence-mysql` 实现 MySQL 8.4 差异化 DDL、索引和查询；领域层和应用层只依赖公共端口，不引用 MySQL 专有实现。
6. **CI 强制校验**：CI 使用 Spring Modulith 与 ArchUnit 校验模块依赖方向与循环引用，违反边界的构建必须失败。

## 4. 替代方案

### 4.1 方案 A：从第一天起采用微服务

将 14 个业务模块拆分为独立可部署服务，每个服务拥有自己的数据库和运行时。

**否决理由**：

- 当前没有任何容量、隔离、合规或组织边界证据表明某个模块需要独立部署；P1 的 1000 并发用户和百万级数据规模在模块化单体配合无状态多副本与单一 MySQL 写入主库下可达成性能目标。
- 微服务将引入服务发现、跨进程通信、分布式事务/编排、独立 schema 同步、独立可观测性和多制品发布等运维开销，远超 P1 团队与运维能力。
- 违反宪章原则 IV“从满足当前质量属性的最简单方案开始”；微服务拆分必须由证据驱动并通过 ADR 记录。

### 4.2 方案 B：单层单体，不建立模块边界

将所有业务逻辑集中在一个无内部边界的大型代码库中，模块间任意调用内部实现。

**否决理由**：

- 违反宪章原则 IV“受治理扩展”：缺乏显式模块边界将使领域包、工作流和受治理扩展无法稳定接入，稳定内核得不到隔离。
- 跨模块直接访问内部表与实现类将使未来按容量或组织边界拆分服务变得不可能或代价极高，破坏“简单演进”的拆分路径。
- 无法满足 `plan.md`“模块只能依赖其他模块公开的应用服务、只读查询接口或领域事件”的明确约束。

### 4.3 方案 C：模块化单体 + 每模块独立数据库

在模块化单体内部为每个业务模块分配独立数据库/schema 作为唯一写入源。

**否决理由**：

- 违反 P1“每部署单一业务写入主库”的约束和单写主权要求。
- 跨模块查询（如多视图、跨项目搜索、统一待办、报表聚合）需要跨库 join 或应用层聚合，引入未经验证的复杂度，且无法依赖单一 MySQL 的事务一致性保证。
- 增加数据源路由、连接预算和事务管理复杂度，与 `plan.md`“在线池总连接预算不超过数据库可用连接的 70%”等容量约束冲突。

## 5. 质量属性影响

| 质量属性 | 影响 |
|---|---|
| **可扩展性（Scalability）** | 应用层无状态，可通过多副本水平扩展承载 1000 并发用户；写入瓶颈由单一 MySQL 主库与连接预算约束，必要时通过读副本 `pdpRead` 缓解只读查询压力。 |
| **可维护性（Maintainability）** | 模块边界由 Spring Modulith 2.1 与 ArchUnit 在 CI 中强制执行；模块只能通过公开应用服务、只读查询端口或领域事件协作，避免跨模块内部实现耦合。 |
| **可测试性（Testability）** | 支持模块级独立测试（Spring Modulith 模块测试），每个用户故事可独立验证领域状态机、持久化契约和模块边界。 |
| **可部署性（Deployability）** | 单一可部署制品，简化构建、发布、回滚和版本管理；后台任务执行器复用同一制品，仅运行配置不同。 |
| **可演进性（Evolvability）** | 通过显式端口与领域事件保留未来拆分路径；当容量或边界证据出现时，可按模块逐个抽取为独立服务而不破坏现有契约。 |
| **可观测性（Observability）** | 单一制品便于统一日志、指标、追踪和审计；模块边界使责任组件定位清晰。 |

## 6. 数据与迁移影响

- P1 每个部署使用单一 MySQL 8.4 LTS 业务写入主库；模块共享 schema 但 **必须** 尊重逻辑表所有权，禁止跨模块直接读写其他模块的内部表。
- Flowable 引擎表使用独立 `workflowEngine` schema/账号/HikariCP 池，与 PDP 业务表隔离，不使用 XA。
- `migrationSource` 与 `migrationTarget` 使用独立 `SqlSessionFactory`、Mapper 扫描包、本地事务管理器、账号和 HikariCP 池；源事务只读，目标批次只在目标本地事务提交。
- MySQL 历史数据迁移采用“冻结旧系统写入—应用最终增量—核对—启用 PDP—旧系统只读”的单向切换，不采用长期双写；PDP 开始接收新写入后，回切旧系统必须先有经过验证的反向同步。
- **未来拆分的数据影响**：若某模块拆分为独立服务，必须先抽取该模块的表、仓储端口与事件契约，并通过迁移窗口完成数据切割与一致性核对；每次拆分必须产出新的 ADR。

## 7. 拆分触发条件 (Split triggers)

当以下任一条件触发时，应启动模块拆分评估并产出新的 ADR：

1. **容量证据**：P1 性能与容量测试或生产监控数据显示，某一模块持续主导 CPU、内存、数据库连接或写入负载，且无法通过水平扩展或索引优化缓解。
2. **隔离与合规**：某模块因数据隔离、合规审计或多租户边界要求，必须与其他模块物理隔离部署。
3. **组织边界**：某模块由独立团队或供应商负责，其开发与发布节奏与平台主制品显著错位，模块边界已无法支撑协作效率。
4. **部署节奏**：某模块的发布频率显著高于或低于平台主制品，且模块化单体的统一发布约束已成为瓶颈。

每次拆分必须：产出新 ADR 记录背景、选项、后果与退出条件；先更新 OpenAPI、事件或领域包契约并通过契约测试，再实现提供方或消费方迁移；保留数据切割、回退点和不可逆点。

## 8. 验证方式

模块化单体边界通过以下自动化手段在 CI 中持续验证：

- **Spring Modulith 模块测试**：`ModuleBoundaryTest` 验证每个模块的公开 API 与内部类型可见性。
- **ArchUnit 依赖规则**：校验模块依赖方向、禁止循环依赖、禁止跨模块直接访问内部表或实现类。
- **契约测试**：OpenAPI、事件契约与领域包 schema 的变更必须先于 Controller、消费者或运行时实现，并通过 Spectral/AJV 校验。
- **模块 API 依赖检查**：模块 API 只能依赖其他模块的公开应用服务、只读查询端口或领域事件，违反时构建失败。
- **持久化契约测试**：每个持久化用户故事交付公共 DDL、仓储端口、Mapper/适配器和 MySQL 8.4 契约测试，验证领域层与应用层不依赖 MySQL 专有实现。

## 9. 复审条件

本 ADR 在以下任一条件触发时复审，并记录继续有效、修订或撤销结论及责任人：

1. **P1 性能与容量测试结果可用时**：若 1000 并发用户、百万级数据或后台作业压测结果表明模块化单体无法满足 SLA/SLO，启动拆分评估。
2. **拆分触发条件命中时**：见第 7 节任一条件触发。
3. **季度治理复核**：依据宪章治理原则，至少每季度复核一次。
4. **重大生产事故或监管变化**：发生重大生产事故、监管变化、产品战略调整或架构边界变化时触发专项复核。
5. **认证数据库变化**：P2 引入 PostgreSQL 适配器或跨数据库切换时，重新评估持久化边界与模块化单体适配器注册机制。

## 10. 追溯

- **宪章原则 IV（稳定内核、受治理扩展与简单演进）**：本 ADR 是“架构 MUST 从满足当前质量属性的最简单方案开始”与“新增服务拆分 MUST 由容量、隔离、合规或组织边界证据驱动，并通过 ADR 记录”的直接落地。
- **`plan.md` 架构与模块分解**：运行时边界、业务模块表与“结构决策：后端先保持单一可部署应用，但按业务能力建立强模块边界；后台任务以同一制品的独立运行配置部署”。
- **`plan.md` 复杂度与例外跟踪**：模块化单体为满足当前范围的最小可行复杂度，无宪章例外。
- **`spec.md` 产品使命与范围**：14 个业务与运行模块 + 持久化 + 工作流 + 标准实施领域包 + 历史迁移构成 P1 业务闭环。
- **`technology-comparison.md`**：P1 仅认证 MySQL 8.4 LTS，数据库独立通过领域/应用端口与适配器边界实现，PostgreSQL 适配器进入 P2。
- **FR 覆盖**：见第 2 节业务模块表的“主要追溯”列，覆盖 FR-003～FR-174 中 P1 范围内全部功能需求。
