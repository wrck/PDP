# ADR 0004：MySQL 历史数据迁移、切换与本地事务边界

## 状态与日期

- **状态**：Accepted
- **日期**：2026-07-17
- **追溯**：宪章原则 VIII「可观测、可恢复与安全迁移」与「当前交付与工程门禁」迁移及多数据源条款；`specs/002-pdp-product/mysql-migration.md`；`specs/002-pdp-product/plan.md`「MySQL 历史数据迁移边界」与「P1 服务等级与容量矩阵」；`specs/002-pdp-product/persistence-design.md` 第 2、9 节；ADR 0003。

## 背景

PDP 必须将现有 MySQL（5.7/8.x）系统的历史数据迁移到 PDP MySQL 8.4 LTS。虽然源与目标均为 MySQL，但「同数据库产品」仅降低物理类型转换成本，不能跳过业务迁移门禁：必须完成统一标识、字段语义、状态机、组织、权限、审计和 schema 版本转换，并经过技术核对与业务核对。搬运成功不等于业务正确。

宪章原则 VIII 要求：数据迁移和数据库切换 MUST 可追溯、可核对、可重放、可暂停，并具有回退点和不可逆点。`plan.md`「P1 服务等级与容量矩阵」为历史迁移与上线切换设定 RTO ≤ 30 分钟、RPO ≤ 5 分钟、单写主权 100%、两次全量彩排和单向切换的强制目标。任何时刻同一生产部署只允许一个业务写入主库；不长期双写。

迁移是独立产品交付流，不是临时 SQL 脚本集合。原始数据、转换数据和目标业务数据 MUST 分层隔离；每条迁移规则版本化，每次运行可重放，每个目标对象可追溯；失败数据进入隔离区，不得静默丢弃、默认填充或绕过平台规则。

## 决策

1. **异构迁移工具只负责可靠提取与传输，不决定 PDP 业务语义。** MySQL Shell 一致性 dump、自建并行 JDBC 或云 DMS 等工具仅完成源快照、行级来源保留和受控传输；PDP 业务语义由版本化映射层与受控装载层负责。
2. **原始 MySQL 数据先进入隔离的迁移暂存区，再由版本化映射转换为 PDP 命令或受控批量装载格式。** 暂存区使用独立 schema 或独立实例，使用与在线应用不同的账号；在线业务账号不得读取原始源数据，迁移账号不得获得平台管理权限。
3. **`migrationSource` 与 `migrationTarget` 使用独立 `SqlSessionFactory`、Mapper 扫描包、本地事务管理器、账号和 HikariCP 池。** 源事务只读；目标批次只在目标本地事务内提交。源读取事务与目标写入事务不得包装为分布式事务；不使用 XA 或跨库两阶段提交。`pdpPrimary` 始终承担在线业务写入主权；`pdpRead` 不参与迁移门禁与核对。
4. **核心对象 MUST 经过必填、唯一、关系、状态、权限和审计校验。** 无法映射的数据进入隔离区，不得静默丢弃。禁止按名称猜测字段语义；客户、合同、项目、阶段、任务、状态、审批、交付件和用户等公共数据口径必须由业务负责人确认。
5. **全量迁移后通过 MySQL binlog 或等价变更源捕获增量。** 正式切换采用单向切换：冻结旧系统写入 → 应用最终增量 → 核对 → 启用 PDP → 旧系统只读。
6. **不长期双写。** PDP 开始接收新写入后，回切旧系统必须先有经过验证的反向同步；否则采用前向修复。P1 默认不建设长期反向同步，正式开放 PDP 写入即视为主要不可逆点。
7. **迁移数据 MUST 保留血缘字段**：`source_system`、`source_table`、`source_key`、`migration_run_id`、`mapping_version`。在线 API MUST NOT 向普通用户暴露内部迁移控制字段，这些字段仅用于迁移控制面、审计和核对。

## 替代方案

- **(a) 长期双写**：拒绝。引入双写一致性、冲突协调和回滚复杂度，违反单写主权 100% 的服务等级目标，长期维护成本与一致性风险高。
- **(b) 不带增量的大爆炸切换**：拒绝。冻结窗口内无法追平存量与新增变更，RPO 风险和停机时间不可接受，违反 RPO ≤ 5 分钟目标。
- **(c) 共享业务数据源进行迁移**：拒绝。业务 Mapper 与迁移 Mapper 共用 `SqlSessionFactory`、事务管理器或账号会模糊持久化上下文与事务边界，违反宪章「多数据源和迁移连接 MUST 使用严格路由、独立凭据、连接预算」和 ADR 0003。
- **(d) 反向同步作为默认回退**：拒绝。反向同步仅在预先建设并验证后可作为受控回退；否则一旦 PDP 开放写入，只能执行前向修复。默认假设存在反向同步会造成回退点越界风险。

## 质量属性影响

- **可靠性**：每批次持久化可重放检查点；幂等批次键恢复；跨源/目标不启用分布式事务；失败数据隔离不丢弃；CDC 位点、应用位点、延迟与失败持续监控。
- **可逆性**：明确可安全回退点（旧系统冻结、最终增量完成但 PDP 尚未开放写入前）和不可逆点（PDP 开放写入后，无验证反向同步时不回切）。不可逆点开放写入 MUST 由上线决策人明确批准。
- **可观测性**：保留迁移血缘（源系统、源表、源主键、批次、映射版本）；监控批次进度、CDC 位点与延迟、隔离数量；按数据源监控 HikariCP active/idle/pending/timeout/acquire duration；达到在线保护阈值时暂停新批次。
- **安全**：迁移账号最小权限、独立凭据、定期轮换；全链路 TLS；非生产数据脱敏；日志不得记录密码、完整身份证件、签名原文或附件内容；导出、下载和人工修复双人复核；迁移完成后撤销源库写权限和临时网络通道。

## 数据与迁移影响

- 建立版本化源到目标映射目录，每个映射项包含 `source_system/schema/table/column`、`source_key`、`target_object/field`、`mapping_type`、`transform_rule`、`default_policy`、`validation_rule`、`owner`、`reconciliation_rule`、`exclusion_reason`。
- 建立数据质量基线：识别无主键表、重复主键/业务键、孤儿外键、循环层级、非法或零日期、`TINYINT(1)` 非 0/1 值、`ENUM/SET` 未知值、非法 JSON、`utf8mb3/latin1` 错误编码、排序规则冲突、软删除与审计字段不完整等。
- 建立全量装载耗时基线和增量方案（MySQL binlog 位点、GTID、CDC 工具选型与权限）。
- 切换窗口评估：包含冻结期、最终增量应用、阻断级核对、启用 PDP 和旧系统只读各阶段时间预算。
- 映射覆盖：字段语义；标识（`AUTO_INCREMENT` 不沿用为平台主键，生成 UUIDv7 并写入 `LegacyKeyMap`，`UNSIGNED` 超范围时使用定点数或文本旧键）；状态（`ENUM/SET` 映射为稳定状态键，未知值阻断或隔离）；组织与归属；权限；附件（`BLOB` 转对象存储，数据库保存哈希、大小和版本元数据）。

## 事务边界

- **源事务只读**：`migrationSource` 使用只读账号、`readOnly=true` 的 HikariCP 池、独立只读会话工厂与事务管理器。
- **目标批次本地事务**：每个批次在单一 `migrationTarget` 本地事务内提交，提交后再持久化可重放检查点；禁止将源读取事务与目标写入事务包装为分布式事务。
- **不使用跨库 2PC**：事务开始后不得切换数据源；数据源路由上下文 MUST 在批次 `finally` 中清理，线程池任务显式传递或重建上下文，禁止 ThreadLocal 泄漏。
- **幂等检查点**：批次以「读取源批次并持久化位点 → 独立目标事务幂等写入 → 核对并推进检查点」执行；重试 MUST 从持久化检查点恢复。
- **核对维度**：按对象类型执行技术核对（数量、主键、引用、金额、时间、权限、附件）与业务核对：
  - 数量：源/目标记录数量与排除数量；
  - 主键：映射唯一性，`LegacyKeyMap` 无冲突；
  - 引用：外键与层级完整性，孤儿关系进入隔离；
  - 金额：`DECIMAL` 精度与舍入，`FLOAT/DOUBLE` 转定点后误差核对；
  - 时间：`DATETIME/TIMESTAMP` 按源业务时区解释再转 UTC，零日期隔离；
  - 权限：工作空间归属、成员角色与数据范围；
  - 附件：数量、大小与 SHA-256 哈希一致。

## 切换门禁与回退点

切换流程为单向序列：

```text
增量追平 → 影响预览 → 写入冻结 → 核对门禁 → 启用 PDP → 旧系统只读
```

Go 条件：阻断问题为 0；最终未应用增量为 0；必迁对象数量、关系、审批链和附件哈希核对通过；登录、权限、核心项目和审批冒烟测试通过；备份、恢复和应急人员就绪；正式冻结前 CDC 延迟连续 30 分钟 ≤ 30 秒。

- **可安全回退点**：旧系统冻结、最终增量完成但 PDP 尚未开放写入前，如核对失败可恢复旧系统写入；冻结期间的队列、人工记录和最终位点 MUST 保留。
- **不可直接回退点**：PDP 开始产生新业务写入后，两系统数据已分叉。此时只有两种受控方案：已建设并验证反向同步回放后回切，或保持 PDP 为主系统执行前向修复。P1 默认不建设长期反向同步，正式开放 PDP 写入即视为主要不可逆点，上线决策人 MUST 明确批准。

正式切换前 MUST 至少完成两次全量彩排加一次增量彩排，覆盖全量快照与装载、持续 CDC、模拟写入冻结、应用最终增量、完整核对、模拟启用 PDP 和回退或前向修复演练；每次记录总耗时、各表吞吐、CDC 峰值延迟、问题数量、人工步骤和改进项。

## 验证方式

- **MigrationControlDatabaseContractTest**：验证迁移控制面数据源路由、独立 `SqlSessionFactory`、Mapper 包隔离、本地事务边界、事务内切换拒绝和上下文清理。
- **MySqlHistoryMigrationResilienceTest**：覆盖乱码数据、非法日期、重复主键、孤儿关系、源中断和目标回滚等失败场景，验证隔离、检查点恢复与幂等重放。
- **MigrationLineageTest**：验证每条迁移记录可追溯到源系统、源表、源主键、批次和映射版本。
- **彩排证据**：至少 2 次全量彩排加 1 次增量彩排，证据保存于 `evidence/us20-migration-rehearsal.md`，包含总耗时、吞吐、CDC 延迟、问题清单和改进项。
- **核对证据**：技术核对与业务核对报告、问题隔离与人工裁定清单。

## 运行手册与业务签字

- 运行手册路径：`docs/runbooks/data-cutover.md`，覆盖全量装载、CDC、冻结、核对、启用 PDP、回退和前向修复各阶段操作、责任人和告警条件。
- 业务签字：正式切换 MUST 由产品、领域/业务验收、架构、质量、运维/SRE 及适用安全/合规责任人完成所需签字；不可逆点开放写入 MUST 由上线决策人明确批准。

## 复审条件

- 每次彩排后；
- 正式切换完成后；
- 源系统 schema、字符集、binlog 配置或 CDC 工具发生变化时；
- 每季度专项复核（与宪章原则 VIII 复核节奏一致）。

复核 MUST 记录继续有效、修订或例外结论及责任人。
