## 1. 宪章目的

本宪章是“项目交付平台”所有产品决策、需求规格、技术设计、开发实现、测试验收和发布运营的最高约束。平台采用 Spec Kit 的规范驱动开发思想：先明确意图和可验证结果，再形成方案与任务，最后实施；代码不是需求的替代品，会议结论也不是可追溯规格的替代品。

当本宪章与需求文档、设计文档、迭代计划或临时决策冲突时，以本宪章为准；如确需偏离，必须按照“治理与修订”章节完成评审、记录和批准。

## 2. 项目使命与愿景

### 2.1 使命

建设一个贯穿项目从需求进入、立项、计划、执行、监控、变更、验收、结项到复盘的统一交付平台，使组织能够用同一套业务语言、数据标准和治理机制管理交付全过程。

### 2.2 愿景

- 管理层可以看到真实、及时、可解释的项目组合状态。
- 项目经理可以用标准流程完成计划、资源、成本、风险、问题、变更和交付物管理。
- 项目成员可以明确知道“当前要做什么、由谁负责、何时完成、完成标准是什么”。
- 客户、供应商及审计人员在授权范围内获得一致、可追溯的交付证据。
- 平台能力可通过标准 API、事件和扩展点与 ERP、财务、HR、采购、代码仓库、CI/CD、协作和身份系统集成。

## 3. 范围与边界

### 3.1 首期核心范围

1. 项目需求与机会接入、评估及优先级管理。
2. 项目立项、项目章程、阶段门和审批。
3. 项目组合、项目群、项目及子项目管理。
4. WBS、里程碑、依赖、基线和进度跟踪。
5. 组织、角色、资源池、资源申请及分配。
6. 预算、成本、工时、合同及采购协同。
7. 风险、问题、行动项、决策和变更控制。
8. 交付物、评审、验收、发布和结项。
9. 文档、知识、复盘和经验沉淀。
10. 通知、待办、报表、仪表板、审计和开放集成。

### 3.2 非目标

- 不在首期重建通用财务总账、薪资、采购执行或源代码托管系统。
- 不以大量客户化分支替代统一产品模型；差异优先通过配置、规则、模板和扩展点实现。
- 不以“功能数量”作为成功标准；未形成闭环、不可验证或缺少责任人的功能不得视为完成。
- 不用平台替代组织治理；平台负责固化、执行和留痕已经明确的治理规则。

## 4. 核心原则

### 原则一：规范是唯一可信源（Specification as Source of Truth）

- 每个业务能力必须始于独立、可评审的 `spec.md`，说明用户、场景、业务价值、边界、异常和验收标准。
- 需求必须使用业务语言描述“需要什么和为什么”，不得用页面字段、数据库表或指定技术方案冒充需求。
- 所有功能必须具有稳定的 Feature ID；规格、计划、任务、代码提交、测试、发布说明和审计证据必须能够追溯到该 ID。
- 口头约定、聊天记录和原型中的隐含行为必须回写规格后才具有效力。

**检查标准：** 若团队无法仅依据规格判断一个功能是否验收通过，该规格不得进入计划阶段。

### 原则二：业务闭环优先于功能堆叠（Business Loop First）

- 每项能力必须定义触发条件、参与角色、输入、状态变化、业务规则、输出、异常处理和闭环证据。
- 任何待办都必须有责任人、期限、状态、完成定义和升级机制。
- 风险、问题、变更、审批、验收等对象不得只记录信息，必须支持从提出到关闭的完整生命周期。
- 先交付可独立产生价值的纵向切片，再扩展更多字段、报表或配置项。

**检查标准：** 每个用户故事必须可独立演示，并产生可查询、可审计的业务结果。

### 原则三：领域模型统一业务语言（Domain-Driven Design）

- 产品、研发、测试和实施团队必须使用统一的领域词汇及定义。
- 项目、阶段、工作包、里程碑、交付物、基线、风险、问题、变更、资源分配、验收等核心概念必须有明确的边界和不变量。
- 聚合根负责维护业务一致性；跨聚合流程通过应用服务、领域事件或流程编排完成。
- 核心业务规则不得散落在前端、报表 SQL、工作流脚本或外部集成中。
- 每个领域对象必须明确归属的限界上下文，禁止创建含义不清的“万能项目表”或“万能状态字段”。

**检查标准：** 进入开发前必须完成领域词汇表、核心对象关系、状态机和关键业务不变量评审。

### 原则四：契约先行、兼容演进（Contract First）

- 外部及跨服务接口必须在实现前定义 OpenAPI、AsyncAPI 或等价的机器可读契约。
- API 必须明确身份、权限、幂等性、分页、错误码、并发控制、版本策略和审计信息。
- 事件必须定义名称、业务含义、生产者、消费者、Schema、顺序、重复投递及失败处理策略。
- 已发布契约默认保持向后兼容；破坏性变更必须版本化，并提供迁移窗口和废弃计划。
- 前后端及集成开发应基于契约并行，契约测试是合并和发布的必要门禁。

**检查标准：** 未通过契约评审与自动校验的接口不得进入联调。

### 原则五：工作流可配置，状态机必须确定（Workflow with Deterministic State）

- 审批、阶段门、变更控制和验收流程应支持按组织、项目类型和金额等条件配置。
- 业务状态必须由领域状态机定义，工作流引擎负责编排，不得成为核心业务事实的唯一存储。
- 每次状态迁移必须定义前置条件、授权角色、业务动作、失败结果、补偿策略和审计记录。
- 长事务采用可恢复、可重试、可补偿的机制；不得依赖无法追踪的人工修数完成流程。
- 进行中的流程定义升级必须有实例迁移、兼容或保留旧版本的明确策略。

**检查标准：** 所有状态迁移均须通过状态机测试，非法迁移必须被拒绝并留下可诊断记录。

### 原则六：安全、租户隔离与审计内建（Security by Design）

- 身份认证使用统一身份源；授权采用 RBAC 与 ABAC 结合，至少考虑租户、组织、项目、数据归属、角色、敏感等级和操作上下文。
- 默认拒绝访问，遵循最小权限、职责分离和临时授权原则。
- 多租户数据必须在请求、缓存、检索、消息、导出、备份和日志全链路隔离。
- 敏感数据在传输和存储时必须加密；密钥、令牌及密码不得进入源码、普通配置或日志。
- 关键业务操作必须形成防篡改审计轨迹，记录操作者、时间、对象、动作、前后值、来源和关联请求。
- 安全、隐私和合规需求与业务功能具有同等优先级，不得延后补做。

**检查标准：** 威胁建模、权限矩阵、越权测试、租户隔离测试和审计验证未通过，不得发布。

### 原则七：质量必须可执行、可证明（Executable Quality）

- 验收标准必须使用 Given/When/Then 或同等可验证形式，覆盖主流程、边界、异常、权限和并发场景。
- 测试遵循风险分层：领域规则以单元测试为主，边界以契约和集成测试为主，关键用户旅程以端到端测试为主。
- 缺陷修复必须先增加可复现失败的测试，再实施修复。
- 核心领域规则、权限决策、财务计算、状态迁移和数据隔离必须自动化测试。
- “开发完成”不等于“交付完成”；只有通过验收标准、质量门禁并形成证据才算完成。

**检查标准：** 每个任务必须关联验收标准和测试；无证据的“已测试”不得作为发布依据。

### 原则八：可观测、可恢复、可运营（Operability First）

- 每项关键能力必须定义 SLI/SLO、容量假设、失败模式、告警条件和运行手册。
- 日志、指标和追踪必须使用统一关联 ID，并能定位到租户、用户、项目、请求和业务操作，同时避免泄露敏感信息。
- 异步任务必须可查询状态、可安全重试、可识别重复、可进入死信处理并支持人工恢复。
- 数据备份必须定期验证恢复能力；灾难恢复目标需明确 RPO 和 RTO。
- 重大故障必须进行无责复盘，并将改进项纳入可追踪计划。

**检查标准：** 关键场景未完成监控、告警、恢复演练和容量验证，不得进入生产。

### 原则九：简单、可演进且避免过度设计（Simplicity and Evolution）

- 优先选择能够满足已确认需求的最简单架构；引入新服务、新中间件或新抽象必须有可量化理由。
- 默认采用模块化边界；只有在独立扩缩容、隔离、团队自治或可靠性收益明确时才拆分微服务。
- 配置优先于硬编码，但配置必须有 Schema、版本、校验、默认值、变更审计和回滚能力。
- 技术债必须显式登记影响、风险、偿还条件和负责人，不得以隐藏复杂度换取表面进度。
- 所有架构决策必须用 ADR 记录背景、选择、替代方案和后果。

**检查标准：** 计划评审须检查是否存在没有当前需求支撑的抽象、服务或基础设施。

### 原则十：用户体验服务于交付决策（Outcome-Oriented UX）

- 页面信息架构必须围绕角色任务和业务对象组织，而不是照搬数据库或部门组织结构。
- 同一业务概念在不同页面必须保持名称、状态、颜色、操作语义一致。
- 列表、详情、工作台和报表必须支持从异常发现到责任定位再到行动闭环。
- 关键流程必须满足可访问性、响应式、错误恢复和大数据量使用要求。
- 高风险操作必须确认影响范围，并提供预览、撤销或补偿能力。

**检查标准：** 核心角色的关键旅程必须经过可用性验证，且能够在规定步骤和时间内完成。

## 5. 规格驱动交付流程

每个 Feature 必须按以下顺序推进，任何阶段不得用后续产物反向掩盖前序决策缺失：

1. **Constitution**：检查需求是否符合本宪章。
2. **Specify**：编写 `spec.md`，明确用户价值、场景、边界、业务规则和验收标准。
3. **Clarify**：消除高影响歧义；未解决项必须明确记录假设、风险和责任人。
4. **Plan**：编写 `plan.md`，确定领域边界、架构、数据、接口、安全、迁移、可观测性和测试策略。
5. **Tasks**：生成 `tasks.md`，任务按可独立验证的用户故事纵向拆分，标明依赖及可并行项。
6. **Analyze**：检查宪章、规格、计划和任务之间的一致性、覆盖度及冲突。
7. **Implement**：按照任务顺序实现，每项提交关联 Feature ID 和 Task ID。
8. **Validate**：执行自动化测试、非功能验证、验收和证据归档。
9. **Release**：完成变更说明、迁移方案、回滚方案、运行手册和发布批准。
10. **Learn**：观察业务指标和运行数据，将反馈转化为新规格或宪章修订建议。

## 6. 规格最小完备标准

每份 `spec.md` 至少包含：

- 背景、问题陈述和预期业务价值。
- 用户角色、利益相关者和权限边界。
- 按优先级排列、可独立验收的用户场景。
- 功能需求及唯一编号。
- 业务规则、计算口径和关键不变量。
- 状态、状态迁移及异常路径。
- 数据输入、输出、保留、归档和审计要求。
- 与外部系统或其他限界上下文的交互。
- 性能、可用性、安全、隐私、合规和可访问性要求。
- 明确的范围外事项、依赖、假设和风险。
- 可度量且与实现方式无关的成功标准。

凡出现“快速、灵活、友好、实时、支持大数据量”等不可度量措辞，必须补充具体口径和阈值。

## 7. 进入开发条件（DoR）

Feature 只有同时满足以下条件才可进入开发：

- 与项目使命、范围和当前路线图一致。
- 用户场景、范围边界和验收标准明确。
- 关键业务规则、领域模型和状态机已评审。
- 权限、租户隔离、审计和数据敏感性已明确。
- API、事件及外部依赖契约已定义或已有版本可复用。
- 数据迁移、兼容性和历史数据影响已评估。
- 非功能指标可测试，测试策略和验收责任人明确。
- 高影响歧义已消除，剩余假设和风险已被接受。
- 任务已按用户价值纵向拆分，依赖关系清晰。

## 8. 交付完成条件（DoD）

Feature 只有同时满足以下条件才可标记完成：

- 所有已批准验收标准通过并保留证据。
- 自动化测试、契约测试、权限测试和回归测试通过。
- 无未批准的严重或高风险缺陷。
- API、事件、数据模型、用户帮助及运行文档与实现一致。
- 审计、日志、指标、追踪和告警已实现并验证。
- 性能、容量、可用性、安全和恢复目标达到规格要求。
- 数据迁移脚本可重复执行，回滚或补偿方案已验证。
- 发布说明、已知限制、运维手册和支持交接已完成。
- 产品负责人及指定业务验收人批准。
- 规格、任务与实际交付结果之间可双向追溯。

## 9. 架构与数据约束

- 系统采用清晰的领域分层或六边形架构，领域层不得依赖 Web、数据库或工作流引擎实现细节。
- 每个业务数据对象必须有权威来源和明确所有者；跨域复制的数据必须标注同步方式及一致性模型。
- 事务边界默认限制在单一聚合或单一服务内；跨边界一致性使用事件、Saga 或明确的补偿机制。
- 关键写操作必须支持幂等、防重复提交和乐观并发控制。
- 报表与分析负载不得无控制地影响核心交易链路。
- 删除策略必须区分逻辑删除、归档、匿名化和物理清除，并符合审计及保留要求。
- 时间统一存储为 UTC，展示时按用户时区转换；金额必须同时记录数值、币种和汇率上下文。
- 所有主数据引用必须定义失效、合并和历史追溯规则。

## 10. 成功指标

平台上线后的指标由业务负责人给出基线和目标值，至少包括：

| 维度 | 指标示例 | 目标原则 |
|---|---|---|
| 采用 | 活跃项目覆盖率、活跃用户率 | 核心项目真实使用，而非重复线下维护 |
| 效率 | 立项周期、审批周期、周报耗时 | 相比基线持续下降 |
| 交付 | 按期里程碑率、按期验收率 | 定义统一且趋势可解释 |
| 治理 | 风险按期处置率、变更合规率 | 关键事项形成闭环和证据 |
| 数据 | 必填完整率、状态及时率、一致性差错率 | 口径明确、责任到人 |
| 质量 | 生产缺陷逃逸率、变更失败率 | 核心流程风险持续降低 |
| 可靠性 | 可用性、P95 响应时间、恢复时间 | 达到对应服务等级目标 |
| 价值 | 预算偏差、资源利用、客户满意度 | 与业务结果关联，不追求虚假精确 |

任何指标必须同时定义名称、业务意义、公式、数据源、统计周期、维度、负责人和防误用说明。

## 11. 决策权与责任

| 角色 | 核心责任 |
|---|---|
| 项目发起人/指导委员会 | 批准使命、范围、重大投入、重大例外和宪章修订 |
| 产品负责人 | 对价值、优先级、规格完整性和业务验收负责 |
| 领域负责人 | 对领域术语、规则、不变量和跨域边界负责 |
| 架构负责人 | 对架构适配性、技术风险、非功能目标和 ADR 负责 |
| 安全与合规负责人 | 对安全、隐私、租户隔离和合规门禁负责 |
| 交付负责人 | 对计划、依赖、质量门禁、发布准备和风险升级负责 |
| 开发团队 | 对实现质量、测试、可观测性和技术文档负责 |
| 业务验收人 | 对场景真实性、验收证据和上线接受负责 |
| 平台运维/SRE | 对容量、可靠性、告警、恢复和生产运营负责 |

## 12. 例外管理

任何不符合本宪章的方案必须提交书面例外，至少说明：

- 违反的原则及具体条款。
- 无法遵循的业务或技术原因。
- 已评估的替代方案。
- 对安全、质量、成本、进度和运营的影响。
- 降低风险的补偿控制。
- 例外有效期、整改计划、责任人和复审日期。

未经产品、架构以及相关安全/合规责任人批准的例外，不得进入生产。永久例外应通过修订宪章或正式架构决策处理，不得无限延期。

## 13. 治理与修订

### 13.1 宪章权威

- 本宪章高于单个 Feature 的规格、计划和任务。
- 所有规格评审、架构评审和发布评审必须显式执行“宪章检查”。
- 发现实现与宪章冲突时，默认修正实现或规格，而不是静默忽略宪章。

### 13.2 修订流程

修订必须包含提案、影响分析、利益相关者评审、批准、版本升级和迁移说明。版本遵循语义化规则：

- **MAJOR**：删除原则、改变治理权责或引入不兼容的根本性要求。
- **MINOR**：新增原则、章节或实质性扩展约束。
- **PATCH**：不改变原意的澄清、文字修正和示例补充。

宪章修改后，必须检查并同步更新所有模板、现行规格、开发计划、质量门禁和团队工作约定。

### 13.3 定期复核

- 每季度至少复核一次宪章适用性。
- 重大架构调整、严重生产事故、监管变化或产品战略变化后必须触发专项复核。
- 复核结果必须记录“继续有效、需要修订或批准例外”，不得仅记录会议已召开。
