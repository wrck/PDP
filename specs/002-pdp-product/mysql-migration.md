# MySQL 历史数据迁移与上线切换方案

## 1. 目标与原则

目标是在可批准的停机窗口内，将现有 MySQL 历史数据转换为 PDP 统一业务模型，并证明完整性、准确性、可追溯性和可恢复性。

原则：

- 迁移是独立产品交付流，不是临时 SQL 脚本集合。
- 现有 MySQL 是迁移源，目标为当前部署选择的 PostgreSQL 18 或 MySQL 8.4 LTS；切换后不长期双写。
- 原始数据、转换数据和目标业务数据分层隔离。
- 每条迁移规则版本化，每次运行可重放，每个目标对象可追溯。
- 搬运成功不等于业务正确，必须执行技术核对和业务核对。
- 失败数据进入隔离区，不能静默丢弃、默认填充或绕过平台规则。

## 2. 总体架构

```text
现有 MySQL
  ├─ 一致性全量快照
  └─ binlog 增量变更
          ↓
迁移原始区
  ├─ 原表快照
  ├─ 增量事件与位点
  └─ 文件/附件清单
          ↓
标准化与映射层
  ├─ 类型转换
  ├─ 数据清洗
  ├─ 新旧标识映射
  ├─ 状态和组织映射
  └─ 问题隔离
          ↓
PDP 受控装载层
  ├─ 目标数据库适配器
  ├─ 核心对象校验
  ├─ 关系与权限校验
  ├─ 附件对象存储
  └─ 审计与迁移血缘
          ↓
核对、切换与只读保留
```

迁移暂存区使用独立数据库 schema 或独立实例，并使用与在线应用不同的账号。在线业务账号不得读取原始源数据；迁移账号不得获得平台管理权限。

应用内动态数据源只负责受控连接路由，不提供跨数据库事务：

- `migrationSource` 只读连接历史 MySQL 或当前迁移源，使用独立 HikariCP 池、只读账号、`SqlSessionFactory`、Mapper 扫描包和事务管理器；
- `migrationTarget` 连接尚未取得在线写入主权的目标库，使用独立 HikariCP 池、写入账号、`SqlSessionFactory`、Mapper 扫描包和本地事务管理器；
- 当前在线业务始终通过 `pdpPrimary` 写入；可选 `pdpRead` 不参与迁移门禁与核对；
- 迁移 Mapper 不得注入在线业务会话工厂，在线业务 Mapper 不得访问迁移源或迁移目标；
- 迁移批次以“读取源批次并持久化位点 → 独立目标事务幂等写入 → 核对并推进检查点”执行，不使用 XA 或跨库两阶段提交；
- 任何迁移事务开始后不得切换数据源，路由上下文必须在批次结束时清理，重试必须从持久化检查点恢复；
- 迁移池连接预算不得挤占在线连接保留量，达到在线池等待或数据库负载阈值时自动限流或暂停迁移。

## 3. 迁移范围盘点

### 3.1 系统与结构

- MySQL 精确版本、部署方式、主从拓扑和备份方式；
- schema、表、视图、主键、唯一键、外键、索引和分区；
- InnoDB/MyISAM 等存储引擎；
- 字符集、排序规则、`sql_mode`、服务器和会话时区；
- 存储过程、函数、事件、触发器和应用侧隐含规则；
- binlog 格式、保留周期、GTID 和 CDC 可用权限。

### 3.2 数据规模

每张表记录量、容量、日增长量、更新/删除频率、大字段比例、附件数量和总容量。记录最大主键、最早/最晚业务时间和预计全量导出时间。

### 3.3 数据质量

- 无主键表和重复主键/业务键；
- 孤儿外键、循环层级和缺失引用；
- 非法或零日期；
- 无符号整数超出 PDP 逻辑类型或目标数据库映射范围；
- `TINYINT(1)` 实际存在非 0/1 值；
- `ENUM/SET` 未知值；
- 非法 JSON、截断文本和二进制混存；
- `utf8mb3`、latin1 或错误编码数据；
- 大小写、重音和尾随空格导致的唯一键冲突；
- NULL、空字符串、0 和“未知”状态混用；
- 软删除记录、历史版本和审计字段不完整。

盘点结果必须形成数据目录、质量报告、责任人和处理截止时间。

## 4. 源到目标映射

每个映射项至少包含：

| 字段 | 说明 |
|---|---|
| `source_system/schema/table/column` | MySQL 来源 |
| `source_key` | 原始主键或组合键 |
| `target_object/field` | PDP 核心对象、扩展字段或动态对象 |
| `mapping_type` | 直接、计算、查表、拆分、合并、枚举或不迁移 |
| `transform_rule` | 版本化转换规则 |
| `default_policy` | 仅在业务批准时使用的默认策略 |
| `validation_rule` | 类型、必填、唯一、关系和状态校验 |
| `owner` | 业务与技术责任人 |
| `reconciliation_rule` | 数量、汇总或业务核对方式 |
| `exclusion_reason` | 不迁移原因及归档方式 |

不允许按名称猜测字段语义。客户、合同、项目、阶段、任务、状态、审批、交付件和用户等公共数据必须由业务负责人确认口径。

## 5. 关键类型转换

| MySQL 情况 | PDP 统一处理 |
|---|---|
| `UNSIGNED INT` | 使用满足 PDP 逻辑范围的定点数，或生成 UUIDv7 并保存旧键映射 |
| `UNSIGNED BIGINT` | 先检查最大值；超过统一有符号范围时用定点数或文本旧键 |
| `TINYINT(1)` | 只有值域确认仅 0/1 后转 Boolean，否则按整数处理 |
| `AUTO_INCREMENT` | 不沿用为平台主键；生成 UUIDv7，旧值进入 `LegacyKeyMap` |
| `DATETIME` | 按源业务时区解释，再转换为 UTC；不得假设已是 UTC |
| `TIMESTAMP` | 核对 MySQL 会话时区转换行为和历史写入方式 |
| `0000-00-00` | 进入问题隔离；按业务规则转 NULL、明确未知值或拒绝 |
| `ENUM` | 映射为稳定状态键或字典项，未知值阻断或隔离 |
| `SET` | 映射为稳定键数组或关系表，禁止直接保留逗号字符串 |
| `JSON` | 先验证并按目标 schema 规范化，再由适配器映射为 PostgreSQL `jsonb` 或 MySQL `JSON` |
| `TEXT` 中 JSON | 不能仅因外观像 JSON 自动转换，需字段级规则 |
| `DECIMAL` | 保持精度和舍入规则，禁止先转浮点 |
| `FLOAT/DOUBLE` | 用于金额、权重时转定点并核对误差 |
| `BIT` | 按位长度映射 Boolean、整数或字节数组 |
| `BLOB` | 文件内容转对象存储，数据库保存哈希、大小和版本元数据 |
| `utf8mb3/latin1` | 显式转 UTF-8；错误字节进入隔离并保留原始副本 |
| 排序规则 | 预先检测大小写、重音和尾随空格引发的唯一冲突 |
| 空字符串 | 按字段语义决定保留或转 NULL，禁止全局统一替换 |

目标为 MySQL 时也不得直接复制旧表结构或存储过程。源库与 PDP 目标库之间仍必须经过统一标识、字段语义、状态、权限、审计和 schema 版本转换；“同数据库产品”只能降低物理类型转换成本，不能跳过业务迁移门禁。

## 6. 迁移阶段

### M0：准备与冻结规则

- 确认范围、责任人、维护窗口和业务签字人；
- 建立源库只读账号、CDC 账号和网络加密；
- 冻结源系统破坏性 DDL；迁移期间的 DDL 必须受控登记；
- 建立迁移版本库、凭据管理、脱敏和审计。
- 为 `migrationSource`、`migrationTarget` 和 `pdpPrimary` 建立独立账号、连接池、`SqlSessionFactory`、Mapper 包、事务管理器、容量预算、路由白名单和故障告警。

### M1：结构与数据画像

- 自动采集结构、数据量、值域、空值率、重复和关系质量；
- 识别高风险表、无主键表、大字段表和高变更表；
- 形成迁移波次和预计时长。

### M2：映射与清洗

- 完成源到目标映射；
- 在源系统或暂存区修复可安全自动处理的问题；
- 无法自动处理的问题进入业务裁定清单；
- 固化映射版本和基准数据集。

### M3：全量迁移

- 获取 MySQL 一致性快照和对应 binlog 位点；
- 并行提取数据到原始区，保留行级来源；
- 按依赖顺序转换并装载工作空间、组织、用户、项目、任务、审批、交付件、附件和审计；
- 装载过程使用幂等批次和新旧标识映射。
- 每批目标写入在单一 `migrationTarget` 本地事务内完成，提交后再持久化可重放检查点；禁止将源读取事务与目标写入事务包装为分布式事务。

MySQL Shell 的一致性 dump 可用于源快照和证据备份；云环境可评估 DMS，全平台环境可使用自建并行 JDBC。工具必须以生产等价数据验证吞吐和大字段行为。[MySQL Shell 导出](https://dev.mysql.com/doc/mysql-shell/8.0/en/mysql-shell-utilities-dump-instance-schema.html)

### M4：增量同步

- 从全量快照位点开始读取 MySQL binlog；
- 将插入、更新、删除转换为幂等迁移事件；
- 按源事务顺序和对象依赖应用；
- 监控读取位点、应用位点、延迟、失败和隔离数量。
- 同时监控迁移源/目标连接池的活跃、空闲、等待、超时和连接失败；达到在线保护阈值时暂停新批次。

可选实现为 Debezium MySQL Connector 或云数据库迁移服务。MySQL binlog 记录已提交的数据与结构变化，CDC 必须在 binlog 清理前持续消费。[Debezium MySQL Connector](https://debezium.io/documentation/reference/stable/connectors/mysql.html)

### M5：技术与业务核对

技术核对：

- 源/目标记录数量和排除数量；
- 主键映射唯一性；
- 外键和层级完整性；
- NULL、日期、数值精度和枚举值分布；
- 附件数量、大小和 SHA-256；
- CDC 位点、重复、失败和积压。

业务核对：

- 工作空间、组织和用户归属；
- 项目父子层级、生命周期和负责人；
- 任务、里程碑、基线和进度；
- 审批节点、办理人、意见和结果；
- 交付件版本、签核、附件和归档；
- 风险、问题和变更关系；
- 抽样项目从创建到归档的业务可读性。

### M6：彩排

至少完成两次生产等价彩排：

1. 全量快照与装载；
2. 持续 CDC；
3. 模拟写入冻结；
4. 应用最终增量；
5. 完整核对；
6. 模拟启用 PDP；
7. 回退或前向修复演练。

每次彩排必须记录总耗时、各表吞吐、CDC 峰值延迟、问题数量、人工步骤和改进项。

### M7：正式切换

```text
发布 Go/No-Go 检查
  → 旧系统进入维护模式并冻结写入
  → 记录最终 binlog 位点
  → 应用所有最终增量
  → 执行阻断级核对
  → 启用 PDP 写入和用户流量
  → 旧系统保持只读
  → 加强监控与业务确认
```

Go 条件：

- 阻断级问题为 0；
- 最终未应用增量为 0；
- 必迁对象数量、关系、审批链和附件哈希核对通过；
- 登录、权限、核心项目和审批冒烟测试通过；
- 备份、恢复和应急人员就绪。

### M8：稳定期与退役

- 旧系统按批准期限只读保留；
- PDP 新写入只进入当前部署选定的目标数据库；
- 每日核对关键汇总，跟踪迁移问题和用户反馈；
- 完成业务、审计、安全和数据保留签字后，按处置流程下线旧系统。

## 7. 回退边界

### 可安全回退点

在旧系统冻结、最终增量完成但 PDP 尚未开放写入前，如核对失败，可恢复旧系统写入。冻结期间的队列、人工记录和最终位点必须保留。

### 不可直接回退点

PDP 开始产生新业务写入后，两个系统的数据已分叉。此时只有两种受控方案：

1. 已建设并验证反向同步，将 PDP 新增变化回放到旧系统后回切；
2. 保持 PDP 为主系统，执行前向修复。

P1 默认不建设长期反向同步，因此正式开放 PDP 写入即视为主要不可逆点。上线决策人必须明确批准。

## 8. 安全与合规

- 迁移账号最小权限、独立凭据和定期轮换；
- 全链路 TLS，暂存文件加密并设置自动过期；
- 非生产数据必须脱敏；
- 日志不得记录密码、完整身份证件、签名原文或附件内容；
- 原始数据、隔离数据和核对报告均纳入访问审计；
- 数据导出、下载和人工修复须双人复核；
- 迁移完成后撤销源库写权限和临时网络通道。

## 9. 迁移运行指标

| 指标 | 上线门禁 |
|---|---|
| 映射覆盖率 | 必迁表和字段 100%，其余有批准排除结论 |
| 阻断问题 | 0 |
| 核心对象来源可追溯率 | 100% |
| 主键映射重复 | 0 |
| 必迁关系缺失 | 0 |
| 附件哈希不一致 | 0 |
| 正式冻结前 CDC 延迟 | 连续 30 分钟 ≤ 30 秒 |
| 最终未应用变更 | 0 |
| 彩排次数 | 至少 2 次 |
| 冻结期间变更丢失 | 0 |
| 重复业务结果 | 0 |

## 10. 交付物

- 源系统与数据目录；
- 数据质量报告；
- 源到目标映射目录；
- 类型与状态转换字典；
- 迁移工具配置和版本；
- 全量与 CDC 运行手册；
- 问题隔离和人工裁定清单；
- 技术核对与业务核对报告；
- 两次彩排报告；
- 正式切换、回退和前向修复手册；
- 旧系统只读保留与退役计划；
- 业务、技术、安全和审计签字记录。
