# 功能规格：PDP 企业级项目交付管理平台

**特性目录**：`specs/002-pdp-product`

**创建日期**：2026-07-16

**状态**：已批准（P1 实现基线）

**规格层级**：L0 产品规格

**主要干系人**：项目发起人、产品负责人、领域负责人、业务验收人、架构负责人、安全与合规负责人、质量负责人、交付负责人、运维/SRE、数据迁移负责人和试点团队

**验收责任**：产品负责人确认范围与成功标准；领域负责人确认术语、规则和状态机；业务验收人确认场景真实性；架构、安全、质量和运维负责人分别确认对应非功能门禁

**输入**：在 PDP 项目内优先建立完整产品规格。`docs/000-init` 下的业务规格、调研和建议为参考材料；后续平台域、能力域和业务域特性必须从本规格自顶向下分解，不得自行扩大、缩减或改变产品边界。

## 1. 产品愿景

PDP 是面向企业项目交付的统一管理与扩展平台。平台以项目计划和交付成果为主线，统一管理工作空间、项目、阶段、任务、检查项、里程碑、交付件、审批、风险、问题、变更、资源、成本、报表和外部协同，并通过可版本化领域包承载标准实施、售前测试、转包代施、割接、验收、维保等行业业务。

PDP 不是单一审批系统，也不是多个业务模块的简单集合。产品必须同时满足：

- 项目团队能够计划、执行、协作、审批、交付和复盘。
- 管理者能够统筹项目组合、资源、成本、风险和经营结果。
- 管理员能够通过受控配置适应组织、客户和行业差异。
- 领域团队能够细化复杂业务，而不修改或绕过平台核心。
- 外部参与者和系统能够在明确授权、审计和失败隔离下协同。
- 关键交付和割接窗口具备可量化的可用性、恢复和降级能力。

### 1.2 产品原则

- **产品规格优先**：本规格是所有后续特性的范围和优先级来源。
- **单一事实源**：不同视图和报表使用同一业务数据与权限口径。
- **交付成果导向**：项目完成必须以任务、检查项和交付件等可验证结果为依据。
- **生命周期解耦**：项目阶段、任务状态和审批状态分别表达生命周期、执行和授权决策。
- **稳定内核、领域扩展**：平台核心保持稳定，行业差异通过领域包和受治理扩展表达。
- **配置可治理**：配置必须可测试、可发布、可审计、可迁移、可回滚。
- **异常优先**：管理页面优先暴露超期、阻塞、风险、冲突和待决策事项。
- **最小授权**：页面、搜索、报表、导出、自动化和集成采用一致权限。
- **核心可降级运行**：非核心能力和外部依赖故障不得拖垮核心交付流程。

## 澄清记录

### 2026-07-16 会话

- Q: 如何平衡统一项目生命周期和领域包阶段定制？ → A: 平台定义统一顶层生命周期，领域包可增加和细化子阶段，但必须映射到顶层阶段。
- Q: 跨工作空间项目采用何种归属和协作模型？ → A: 项目仅归属一个主工作空间，其他工作空间通过显式、限时、可撤销的授权参与协作。
- Q: 跨领域项目采用何种统一进度口径？ → A: 以里程碑权重和必需产出状态计算，领域包可配置权重，但必须满足平台归一化和可追溯校验。
- Q: 公共业务数据与领域字段如何划分？ → A: 平台维护统一核心字段和语义，领域包可增加扩展字段与关系，但不得重复定义或改变核心字段含义。
- Q: P1 首个可上线版本采用何种业务交付边界？ → A: 交付平台基础和标准实施领域包的端到端完整闭环，其他行业领域包进入后续阶段。
- Q: 现有 MySQL 历史数据如何进入新平台？ → A: 作为 P1 上线独立迁移工作流处理，必须完成源数据盘点、语义映射、全量与增量迁移、业务核对、切换和回退演练，不允许未经校验直接写入平台核心数据。
- Q: PDP 是否绑定单一数据库产品？ → A: 不绑定。P1 将 PostgreSQL 和 MySQL 作为首批同等级认证数据库，同一版本制品可在部署时选择其中一种；数据库间切换必须通过受控迁移、核对和切换完成，不支持生产运行中无迁移的热切换。

## 2. 产品范围与优先级

### 2.1 P1 — 产品基础与标准交付最小可行版本

- 工作空间、组织、成员、角色和数据隔离。
- 项目模板、领域包、项目、阶段和主子项目。
- 任务、子任务、检查项、里程碑、依赖和计划基线。
- 交付件版本、审批、签核、发布和归档。
- 风险、问题、变更和统一审批中心。
- 列表、看板、日历、时间线、项目概览和工作台。
- 核心旅程的键盘操作、清晰焦点、非颜色状态表达和自动化可访问性检查。
- 自定义对象、字段、关系、表单、页面、流程、规则和权限。
- 通知、活动记录、搜索、审计、数据保护和备份恢复。
- 标准项目交付领域包，覆盖工前、计划、方案、部署、验收和归档。
- 现有 MySQL 历史数据的盘点、清洗、映射、迁移、核对和上线切换。
- PostgreSQL 与 MySQL 的同等级部署支持、兼容性验证和受控数据库切换。

P1 的完成信号不是单独交付平台配置器，而是使用标准实施领域包完成至少一个真实或等价试点项目从创建、计划、执行、交付、审批、验收到归档的端到端闭环。售前测试、转包代施、割接、回访和维保领域包不属于首个可上线版本的强制范围。

### 2.2 P2 — 企业管理与行业扩展

- 项目组合、目标、管理驾驶舱和跨项目报表。
- 资源、技能、产能、工时、预算、成本和利润分析。
- 代理商、客户和现场工程师的轻量入口及弱网作业。
- 开放集成、事件订阅、运行监控和人工补偿。
- 售前测试、转包代施、割接、回访和维保领域包。
- 客户资产、设备、合同、采购、故障、备件等企业数据协同。

### 2.3 P3 — 高级与可选能力

- 国际化、跨时区、屏幕阅读器深度优化和更广泛的无障碍能力。
- 高级资源优化、情景模拟和智能排期。
- AI 项目摘要、行动项、风险建议和授权知识辅助。
- 设备巡检、配置解析和 AI 故障诊断作为独立产品或领域扩展详细定义。
- 商业 SaaS 所需的套餐、计费、配额和租户运营另立产品规格。

### 2.4 范围外

- 具体语言、前端框架、ORM、数据库驱动、中间件、部署平台和代码组织；数据库兼容范围属于产品能力，不在此排除。
- 完整替代即时通讯、办公文档、财务核算、CRM、采购、设备或故障系统。
- 在本规格中展开所有行业字段、页面和流程细节。
- 允许客户通过私有修改平台核心代码实现定制。

## 3. 自顶向下分解治理

后续规格按以下层级分解：

| 层级 | 规格 | 责任范围 |
|---|---|---|
| L0 | PDP 产品规格 | 产品愿景、完整范围、优先级、统一对象和质量目标 |
| L1 | 平台域规格 | 工作空间、项目执行、领域配置、权限数据、协同集成、分析运营等边界 |
| L2 | 能力域规格 | 任务、交付件、审批、资源、报表、自动化等独立能力 |
| L3 | 业务领域包规格 | 标准实施、售前、转包、割接、验收、维保等行业业务 |
| L4 | 客户差异规格 | 在允许覆盖范围内定义客户或工作空间差异 |

所有子规格 MUST：

- 引用本规格中对应 用户故事、FR 和 成功标准。
- 明确其所属层级、上位规格和不在范围内容。
- 只能细化验收条件和业务模型，不得改变上位优先级或核心语义。
- 如需扩大产品边界，必须先修订本产品规格并完成影响评审。
- 保持项目、任务、交付件、审批、权限、审计等术语一致。
- 将跨域共享能力归属到唯一平台或能力域，避免重复建设。

### 3.1 统一术语、状态机与关键不变量

规格、界面、接口、事件、报表和审计统一使用本规格及 `data-model.md` 定义的对象名称和状态。工作流、
审批和自动化只负责协调，核心业务事实由对象状态和版本记录保存。

| 对象 | 规范状态与主路径 | 关键前置条件 | 非法或冲突结果与证据 |
|---|---|---|---|
| 工作空间 | 草稿 → 已启用 → 已停用 → 已归档 | 归档前无活动项目、未完成迁移和有效跨空间授权 | 拒绝操作，列出阻断对象并记录操作者、原因和对象版本 |
| 领域包版本 | 草稿 → 校验中 → 待审核 → 已发布 → 已弃用 → 已退役；校验失败进入已拒绝后返回草稿 | 发布前通过结构、引用、状态、规则、权限、迁移和回滚校验，设计与发布职责分离 | 保持原生产版本，返回全部冲突和失败校验项并形成发布证据 |
| 项目 | 规划前 → 规划中 → 执行中 → 验收中 → 服务中 → 已关闭；可进入已暂停或已取消 | 顶层状态与领域子阶段唯一映射，推进和受控回退满足权限、版本和完成条件 | 拒绝推进或回退，展示缺失事项、冲突版本和授权原因 |
| 阶段 | 未开始 → 可开始 → 进行中 → 已完成；可进入已阻塞、已暂停或已取消 | 进入、退出、必需任务、检查项、交付件和审批条件全部满足 | 状态不变，列出未满足条件并记录规则版本 |
| 任务 | 草稿 → 就绪 → 进行中 → 已阻塞 → 已完成 → 已归档；可取消，解除阻塞回到前一可执行状态 | 负责人、权限、检查项、依赖和对象版本有效 | 拒绝非法迁移或并发覆盖，返回阻断项及当前版本 |
| 里程碑与基线 | 里程碑：计划中 → 有风险/已达成/已错过/已取消；基线：草稿 → 待批准 → 已批准 → 已取代 | 权重归一、必需产出满足；基线变更需审批并保存差异 | 不计入完成或不替换基线，保留原值、建议值和原因 |
| 交付件 | 草稿 → 审核中 → 已批准 → 已发布 → 已归档；退回进入已拒绝并创建修订 | 内容版本、必填元数据、审批、签核和阶段权限有效 | 禁止覆盖已发布版本，保留原版本和拒绝原因 |
| 审批 | 草稿 → 待处理 → 已通过/已拒绝/已撤回/已取消 | 节点办理人、当前权限、业务对象版本和回写条件有效 | 不回写业务状态，保留审批轮次、冲突对象和失败原因 |
| 风险、问题和变更 | 打开 → 分析中 → 处理中 → 已解决 → 已关闭；可记录原因后重开或取消；变更另有待审批、已批准、应用中、已应用和应用失败 | 责任人、影响分析、审批和应用对象版本有效 | 不覆盖原计划或基线，形成差异、补偿和重试记录 |
| 后台作业 | 排队中 → 运行中 → 已完成；可暂停、失败或取消 | 幂等键、执行身份、范围、资源预算和检查点有效 | 保留检查点、失败明细和可安全重试/人工补偿入口 |
| 历史迁移与数据库切换 | 盘点 → 映射 → 彩排 → 就绪 → 切换中 → 稳定观察 → 已完成；可暂停、取消或判定不切换 | 映射批准、全量与增量追平、核对、冻结、单写主权和 Go/No-Go 全部通过 | 未越过回退点时回退；越过后仅允许已验证反向同步或前向修复，完整记录决策证据 |

以下不变量适用于全部 P1 能力：

- 每个业务对象具有唯一工作空间归属、稳定标识、当前版本和权威状态。
- 状态变化、权限判定、关键审计及事务事件登记在同一业务提交中形成一致结果。
- 工作空间、对象、字段和操作权限默认拒绝；无权与不存在不得通过响应差异泄露。
- 工作流实例、通知、搜索、报表和缓存不得成为业务状态、权限或审批结论的唯一事实源。
- 已发布配置、交付件和已批准基线不得原位覆盖；修订必须形成新版本或差异记录。
- 所有非法迁移、并发冲突和补偿操作必须具有稳定原因分类、下一步建议和关联证据。

## 4. 用户场景与测试（必需）

### 用户故事 1 - 工作空间与组织治理（优先级：P1）

平台管理员创建工作空间并配置组织、成员、角色和数据范围，使事业部、区域、客户或代理商共享平台能力，同时保持配置和业务数据隔离。

**优先级理由**：工作空间是身份、权限、配置和数据边界。

**独立测试**：创建两个工作空间及各自项目、模板和成员，验证普通成员不能查询、搜索、导出或下载另一工作空间数据。

**验收场景**：

1. **给定** 管理员具有平台权限，**当** 创建工作空间并指定负责人，**则** 该空间拥有独立成员、角色、模板、配置和数据范围
2. **给定** 用户仅属于工作空间 A，**当** 请求工作空间 B 的对象或附件，**则** 系统拒绝并记录审计事件

---

### 用户故事 2 - 领域包与深度定制（优先级：P1）

领域管理员通过领域包扩展允许扩展的平台对象或创建新业务对象，配置关系、页面、状态、流程、规则、权限和集成，并以受控方式发布。

**优先级理由**：平台必须支持复杂业务细化，不能只允许修改项目字段和固定流程。

**独立测试**：创建网络设备割接领域包，扩展项目和交付件并新增割接单、步骤和资源对象，完成测试、独立审核和发布。

**验收场景**：

1. **给定** 声明式配置能够表达业务，**当** 管理员设计领域包，**则** 无需修改平台核心即可创建对象、页面、流程和规则
2. **给定** 专业算法无法由声明式配置合理表达，**当** 注册受治理领域扩展，**则** 扩展受版本、权限、隔离、失败和审计规则约束
3. **给定** 设计者提交候选版本，**当** 独立发布者完成测试和影响审核，**则** 才能发布生产版本

---

### 用户故事 3 - 项目模板与项目创建（优先级：P1）

管理员维护项目模板，项目经理从模板创建项目并自动获得阶段、任务、里程碑、检查项、交付件和审批计划。

**优先级理由**：模板将平台能力转换为可复用的项目执行方案。

**独立测试**：从标准实施模板创建项目，验证默认对象、负责人规则、工期和交付件准确生成。

**验收场景**：

1. **给定** 用户有项目创建权限，**当** 选择模板并提交必填信息，**则** 系统生成唯一项目及完整默认计划
2. **给定** 模板发布新版本，**当** 创建新项目，**则** 可使用新版本且存量项目保持原版本快照

---

### 用户故事 4 - 项目生命周期与主子项目（优先级：P1）

项目经理维护项目目标、范围、客户、合同、成员和生命周期；大型项目可拆分子项目并按权限汇总。

**优先级理由**：项目是平台承载交付结果的核心聚合对象。

**独立测试**：创建主项目和两个区域子项目，分别执行并验证主项目汇总、权限和关闭条件。

**验收场景**：

1. **给定** 当前阶段退出条件未满足，**当** 用户推进阶段，**则** 系统阻止并列出未完成事项
2. **给定** 子项目尚未达到关闭条件，**当** 申请关闭主项目，**则** 系统拒绝并展示相关子项目

---

### 用户故事 5 - 任务、检查项与团队协作（优先级：P1）

项目成员将工作拆分为任务和子任务，设置负责人、计划、优先级、工作量和检查项，通过评论、附件和活动记录协作。

**优先级理由**：任务是项目从计划落地到个人执行的基本单元。

**独立测试**：创建父子任务及强制检查项并协作完成，验证完成条件、活动记录和项目进度更新。

**验收场景**：

1. **给定** 任务存在未完成强制检查项，**当** 负责人完成任务，**则** 系统阻止并提示缺失项
2. **给定** 子任务进度变化，**当** 查看父任务和项目概览，**则** 系统按统一规则汇总

---

### 用户故事 6 - 里程碑、依赖和计划基线（优先级：P1）

项目经理设置里程碑、任务依赖和计划基线，识别循环依赖、日期冲突和计划偏差。

**优先级理由**：项目计划必须表达关键节点、前后约束和批准基线。

**独立测试**：建立依赖计划并保存基线，调整关键任务后验证受影响事项和偏差。

**验收场景**：

1. **给定** 新依赖形成闭环，**当** 保存计划，**则** 系统拒绝并指出闭环路径
2. **给定** 已批准基线，**当** 授权用户修改关键日期，**则** 系统展示差异并记录原因

---

### 用户故事 7 - 交付件全生命周期（优先级：P1）

项目团队创建、提交、审核、签核、发布、引用和归档交付件，并保留版本和历史。

**优先级理由**：企业交付最终以受控成果而不是流程状态衡量。

**独立测试**：提交实施方案 v1，经退回后发布 v2，验证版本、意见、签名、权限和归档。

**验收场景**：

1. **给定** 已发布交付件需要修订，**当** 上传新内容，**则** 系统创建新版本且不覆盖原版本
2. **给定** 必需交付件未批准，**当** 完成阶段，**则** 系统阻止并列出缺失项

---

### 用户故事 8 - 统一审批中心（优先级：P1）

审批人统一处理项目、任务、交付件、风险、问题、变更、资源和成本审批。

**优先级理由**：审批是横向授权能力，不应与单一阶段或对象绑定。

**独立测试**：发起交付件和变更审批，验证节点、权限、意见、超时和业务状态回写。

**验收场景**：

1. **给定** 审批人无敏感字段权限，**当** 打开审批，**则** 仅展示完成决策所需的授权信息
2. **给定** 审批退回后重新提交，**当** 开始新一轮审批，**则** 保留全部前序记录

---

### 用户故事 9 - 多视图项目工作区（优先级：P1）

用户通过概览、列表、看板、日历、时间线、交付件、风险问题和活动页查看同一项目。

**优先级理由**：不同角色需要不同视角，但数据和权限必须一致。

**独立测试**：分别在看板、时间线和列表修改任务，验证其他视图同步且权限相同。

**验收场景**：

1. **给定** 任一视图修改任务，**当** 切换视图，**则** 状态、负责人和日期保持一致
2. **给定** 用户保存个人视图，**当** 再次进入，**则** 恢复布局且不影响他人
3. **给定** 用户仅使用键盘且无法依赖颜色辨识状态，**当** 完成查看任务、更新进度、上传交付件和处理待办，**则** 焦点顺序清晰、控件可操作且状态具有文字或图形说明

---

### 用户故事 10 - 风险、问题与变更控制（优先级：P1）

项目团队识别风险、登记问题并通过变更请求调整范围、时间、资源、成本或基线。

**优先级理由**：项目偏差必须独立管理，不能直接覆盖原计划。

**独立测试**：将风险转为问题并发起变更，验证影响、审批、关联和前后差异。

**验收场景**：

1. **给定** 风险已经发生，**当** 转为问题，**则** 系统继承背景并保留双向关联
2. **给定** 变更影响已批准基线，**当** 审批通过，**则** 按批准内容更新并保留差异

---

### 用户故事 11 - 权限、审计与数据生命周期（优先级：P1）

管理员配置功能、数据、对象、字段和操作权限；审计人员追踪业务和配置变化；数据负责人执行保留、归档和处置。

**优先级理由**：平台包含合同、客户、设备、签字、成本和交付文件等敏感数据。

**独立测试**：配置内部、代理商和客户权限，执行修改、下载、导出和归档，验证权限和审计。

**验收场景**：

1. **给定** 代理商访问其他代理商或成本数据，**当** 请求页面、搜索或导出，**则** 系统拒绝并审计
2. **给定** 数据达到保留期限，**当** 归档或处置，**则** 遵循审批和法律保留约束

---

### 用户故事 12 - 项目组合、目标与驾驶舱（优先级：P2）

管理者通过项目组合和角色驾驶舱查看目标、健康度、进度、里程碑、资源、成本、风险和交付。

**优先级理由**：企业管理需要从单项目执行上升到跨项目决策。

**独立测试**：建立多个项目的组合，验证异常优先展示、指标口径、权限和明细钻取。

**验收场景**：

1. **给定** 组合包含延期和高风险项目，**当** 打开驾驶舱，**则** 异常优先且可钻取
2. **给定** 用户无权访问部分项目，**当** 查看组合，**则** 聚合和明细均排除无权数据

---

### 用户故事 13 - 资源、工时与成本（优先级：P2）

资源经理维护技能、区域和产能，项目经理管理工作量、工时、预算和实际成本。

**优先级理由**：进度决策必须考虑资源可用性和项目经济性。

**独立测试**：制造资源冲突和成本超阈值，验证负荷、偏差和预警。

**验收场景**：

1. **给定** 成员负荷超过产能，**当** 查看资源视图，**则** 标识时段、项目和任务
2. **给定** 实际成本超过阈值，**当** 查看项目，**则** 展示偏差和处理提醒

---

### 用户故事 14 - 基础搜索与站内通知（优先级：P1）

项目成员在权限范围内跨项目、跨对象搜索，并通过站内通知中心接收核心待办、审批、任务和交付提醒。

**优先级理由**：基础搜索和站内通知是 P1 日常执行闭环的一部分，不能等到高级自动化和报表阶段。

**独立测试**：创建跨项目任务、审批和交付件，验证搜索结果、稳定排序、权限过滤、通知生成、已读状态和失败重提。

**验收场景**：

1. **给定** 用户仅能访问部分项目，**当** 跨项目搜索，**则** 结果只包含有权对象且打开结果时再次校验权限
2. **给定** 相同核心事件重复投递，**当** 生成站内通知，**则** 不产生重复通知且失败可重提

---

### 用户故事 15 - 外部参与者与移动现场作业（优先级：P2）

代理商、客户和现场工程师通过受限入口接单、采集现场数据、上传交付物、查看进度和签核。

**优先级理由**：项目交付跨越企业内外部和弱网现场。

**独立测试**：代理商工程师在弱网采集照片和检查项并同步，客户查看授权进度并签核。

**验收场景**：

1. **给定** 网络中断，**当** 采集允许离线的数据，**则** 安全暂存并在联网后可控同步
2. **给定** 客户访问外部入口，**当** 查看项目，**则** 仅展示授权信息和签核事项

---

### 用户故事 16 - 开放集成与企业协同（优先级：P2）

外部系统在授权范围内交换客户、合同、采购、设备、审批、故障和备件等数据；管理员管理失败与补偿。

**优先级理由**：PDP 是交付协同中枢，但不能替代或依赖所有外部系统内部能力。

**独立测试**：模拟外部系统正常、重复、乱序和不可用场景，验证去重、重试、补偿和审计。

**验收场景**：

1. **给定** 非关键外部系统不可用，**当** PDP 产生待同步数据，**则** 核心业务继续且事项可追踪
2. **给定** 调用方超出授权，**当** 请求数据或动作，**则** 系统拒绝并记录

---

### 用户故事 17 - 行业业务领域包（优先级：P2）

领域团队基于平台标准包定义标准实施、售前测试、转包代施、割接、验收回访和维保领域包。

**优先级理由**：行业业务必须由统一平台能力承载，同时保留深度业务细化空间。

**独立测试**：为六类业务各创建样例领域包，验证专有对象、流程、页面、规则和集成能够独立运行并共享平台能力。

**验收场景**：

1. **给定** 行业业务需要专有对象，**当** 设计领域包，**则** 可扩展核心对象或创建新对象而不修改平台核心
2. **给定** 客户存在差异，**当** 创建工作空间客户包，**则** 可在允许范围显式覆盖并保持上游来源

---

### 用户故事 18 - 高可用与业务连续性（优先级：P1）

用户在核心项目执行和割接窗口持续使用平台；非核心能力或外部依赖故障时，核心能力可降级并按目标恢复。

**优先级理由**：关键交付窗口不可用会影响现场实施和客户业务。

**独立测试**：模拟核心组件、报表、搜索、扩展和外部系统故障，验证告警、降级、恢复和数据一致性。

**验收场景**：

1. **给定** 报表、AI 或非关键集成异常，**当** 执行核心项目操作，**则** 核心操作继续或提供明确降级
2. **给定** 核心服务故障，**当** 启动恢复，**则** 在恢复目标内恢复且数据丢失不超目标

---

### 用户故事 19 - 国际化、高级无障碍与 AI 辅助（优先级：P3）

用户按语言和时区使用平台；在 P1 核心可访问性基线上进一步支持屏幕阅读器和国际化辅助；AI 提供摘要、行动项和风险建议但不替代用户决策。

**优先级理由**：这些能力提升全球协作和效率，但不得成为核心流程依赖。

**独立测试**：切换语言和时区，使用键盘完成核心操作，并在 AI 不可用时人工完成相同流程。

**验收场景**：

1. **给定** 用户切换时区，**当** 查看同一里程碑，**则** 显示本地化但业务时间点不变
2. **给定** AI 生成建议，**当** 用户未确认，**则** 不改变业务数据或触发审批

---

### 用户故事 20 - MySQL 历史数据迁移与上线切换（优先级：P1）

数据负责人和上线团队将现有 MySQL 中的项目、任务、交付件、审批、附件及关联历史安全迁移到 PDP，并能证明迁移结果完整、准确、可追溯和可回退。

**优先级理由**：历史数据是平台上线和业务连续性的前提；迁移失败、语义错误或关系缺失会直接破坏用户信任和项目执行。

**独立测试**：使用脱敏的生产等价数据完成至少两次全量迁移和彩排切换，验证映射、增量追平、业务核对、权限、附件、审计和回退边界。

**验收场景**：

1. **给定** MySQL 源数据存在类型差异、重复、孤儿关系或非法日期，**当** 执行迁移预检，**则** 系统生成可定位的问题清单、处理规则和阻断级别，不直接污染目标数据
2. **给定** 全量迁移后源系统仍有业务写入，**当** 执行增量同步和最终冻结，**则** 目标数据追平到批准切换点并通过业务核对后才开放 PDP
3. **给定** 切换校验未通过，**当** 尚未越过回退点，**则** 按已演练方案恢复原系统写入且不丢失冻结期间变更
4. **给定** 用户查看迁移后的对象，**当** 追溯数据来源，**则** 可定位原系统、原表、原主键、迁移批次和转换规则

### 用户故事 21 - 多数据库部署与受控切换（优先级：P1）

平台运维团队可以使用同一版本的 PDP 制品部署到已认证的 PostgreSQL 或 MySQL 环境，并在业务要求、运维策略或基础设施变化时，将平台数据受控迁移到另一种已认证数据库，而不改变产品功能、接口语义和业务数据口径。

**优先级理由**：数据库锁定会限制私有化交付、客户基础设施适配、历史系统迁移和长期演进；未经治理的运行时切换又会破坏事务、一致性和恢复边界。

**独立测试**：分别在 PostgreSQL 和 MySQL 上执行同一套安装、数据库变更、契约、权限、核心业务、迁移、性能、备份恢复和端到端测试；再以生产等价数据完成 PostgreSQL → MySQL 与 MySQL → PostgreSQL 双向迁移切换演练。

**验收场景**：

1. **给定** 运维人员持有同一版本应用制品，**当** 通过部署配置选择 PostgreSQL 或 MySQL，**则** 系统无需修改业务代码或重新构建即可完成初始化并提供相同 P1 功能
2. **给定** 领域包包含扩展字段、关系、搜索和报表，**当** 分别运行在 PostgreSQL 与 MySQL，**则** API、权限、状态、结构化过滤、稳定排序、分页、精度和审计语义一致，全文搜索使用同一平台分析器和结果规则
3. **给定** 数据库版本、字符集、时区、存储引擎或必要能力不符合认证基线，**当** 应用启动或执行升级预检，**则** 系统明确拒绝启动或升级并报告不兼容项
4. **给定** 平台需要从一种已认证数据库切换到另一种，**当** 完成导出、装载、增量追平、业务核对、冻结和切换，**则** 仅在所有门禁通过后启用目标库，且保留明确回退或前向修复边界

---

### 用户故事 22 - 高级自动化、报表与多渠道订阅（优先级：P2）

管理员配置版本化自动化规则、统计报表和多渠道订阅，使重复业务动作和管理汇总能够在受控执行身份下自动完成。

**优先级理由**：这些能力提升运营效率，但 P1 可以通过基础搜索、站内通知和人工流程完成核心交付。

**独立测试**：配置超期规则、周期报表和外部通知渠道，验证幂等、递归防护、权限、失败重试、订阅和人工补偿。

**验收场景**：

1. **给定** 相同事件重复到达或规则相互触发，**当** 自动化执行，**则** 不产生重复业务结果并停止递归链路
2. **给定** 外部通知或报表分发渠道不可用，**当** 执行订阅，**则** 核心事务不回滚且失败可追踪、重试和补偿

|---|---|---|---|---|---|
| US1 工作空间治理 | 平台管理员、空间负责人、安全审计人 | 新组织接入、成员或协作申请；组织、角色、范围和期限 | 工作空间和授权按状态机生效；唯一主工作空间、默认拒绝、限时授权 | 可用工作空间、成员与授权清单；创建、到期、撤销和越权审计 | 重复组织、越界授权或撤权冲突被拒绝；恢复原授权版本或重新审批 |

| US3 模板与项目创建 | 模板管理员、项目经理 | 已发布模板、项目主工作空间和必填项目资料 | 模板版本冻结；项目和默认计划原子创建且幂等 | 唯一项目、阶段、任务、里程碑、检查项、交付件和审批计划 | 创建失败不留部分项目；重复请求返回原结果；错误模板可更正后重试 |
| US4 项目生命周期 | 项目经理、项目成员、审批人 | 项目范围变化、阶段完成或关闭申请 | 顶层生命周期与领域子阶段唯一映射；推进、回退和关闭满足完成条件 | 当前状态、阻断项、主子项目汇总和完整历史 | 非法推进、子项目未完成或并发冲突被拒绝；受控回退保留原因 |
| US5 任务协作 | 项目成员、负责人、关注人 | 任务创建、指派、进度、检查项、评论或附件变化 | 任务状态受检查项、依赖、权限和版本控制 | 可执行任务、活动时间线、进度贡献、通知和审计 | 重复提交幂等；冲突显示差异；失败附件隔离且不改变任务事实 |
| US6 计划与基线 | 项目经理、计划审批人 | 里程碑、依赖、权重、基线或日期调整 | 循环依赖被拒绝；权重为 100%；基线批准后不可原位覆盖 | 依赖图、批准基线、偏差、进度贡献和阻塞解释 | 冲突或循环不保存；人工进度调整需审批、期限和补偿恢复 |
| US7 交付件 | 提交人、审核人、签核人、归档人 | 文件和元数据提交、审核、签核、发布或归档 | 已发布版本不可覆盖；签核绑定内容哈希和版本 | 可追溯交付件版本、审批签核、归档清单和下载审计 | 病毒、越权或签名过期被拒绝；退回创建修订，原版本保持可追溯 |
| US8 统一审批 | 发起人、审批人、委托人、业务对象负责人 | 可审批对象、审批定义和当前业务版本 | 审批轮次独立；办理时复核权限；回写必须匹配对象版本 | 节点、意见、附件、动作、终态和回写证据 | 重复点击幂等；撤权或版本冲突不回写，支持退回、重提和失败补偿 |
| US9 多视图工作区 | 项目经理、成员、观察者 | 项目上下文、视图偏好、筛选和更新动作 | 所有视图复用同一状态、权限、排序和版本规则 | 一致的概览、列表、看板、日历、时间线和个人/角色视图 | 无权数据被裁剪；并发编辑展示冲突；失效视图回退到安全默认值 |
| US10 风险问题变更 | 项目团队、变更负责人、审批人 | 风险触发、问题发生或计划变更申请 | 风险可转问题；变更先分析审批再应用；重开、取消和关闭必须记录原因 | 风险矩阵、问题结论、变更差异、审批和应用结果 | 部分失败停止后续应用并保留原基线；支持幂等重试、回滚或人工补偿 |
| US11 权限审计生命周期 | 安全管理员、数据负责人、审计人 | 授权、撤权、导出、保留、法律保留或处置申请 | 默认拒绝；临时授权到期；处置受法律保留和审批约束 | 授权决策、审计链、导出证据、归档和处置结果 | 越权、过期授权和受保护数据处置被拒绝；撤销地址并隔离未完成导出 |
| US14 搜索通知 | 项目成员、待办处理人 | 有权对象变化、核心事件或用户搜索 | 投影最终一致但打开时实时复核；通知按事件幂等 | 稳定搜索结果、站内通知、已读状态、对象链接和失败明细 | 撤权后移除结果；重复事件不重复通知；失败可重提或人工补偿 |
| US18 业务连续性 | 用户、值守人员、运维/SRE | 依赖故障、容量阈值、告警或恢复操作 | 核心与非核心服务按目录降级；恢复满足 RTO/RPO | 告警、降级状态、恢复记录、可用性日报/月报和复盘改进项 | 非关键故障不回滚核心事实；重试、死信、切换或人工补偿均可追踪 |
| US20 历史迁移 | 数据负责人、迁移执行人、业务签字人 | 源库盘点、映射批准、批次执行和切换申请 | 盘点、映射、彩排、追平、核对、冻结和切换按门禁推进 | 映射目录、问题清单、核对报告、来源追踪和 Go/No-Go 证据 | 问题隔离、断点重跑；未越过回退点可恢复旧系统，越过后前向修复 |
| US21 多数据库切换 | 运维/SRE、数据库管理员、业务签字人 | 部署选择、认证预检或数据库切换申请 | 同一时刻单写；全量、增量、核对、冻结、切换和稳定观察 | 认证报告、标准迁移包、语义一致性报告、切换和回退证据 | 环境不兼容快速失败；门禁未过不启用目标库；禁止连接串热切和双主 |

**逐故事验收责任与证据约定**：上述每个 P1 故事均由产品负责人确认范围和成功标准、领域负责人确认
术语/规则/状态机、业务验收人确认场景与业务结果、质量负责人确认测试和证据完整性；安全、架构和运维
负责人按故事风险参与签字。每个故事的证据保存在 `specs/002-pdp-product/evidence/us{故事编号}-*.md`，
准确文件名和生成任务以 `tasks.md` 对应故事阶段的证据任务为准。

### 边界场景

- 工作空间、组织或领域包越界访问必须拒绝并审计。
- 模板、状态或流程存在不可达状态、循环或无责任人时禁止发布。
- 领域包删除被页面、规则、报表、集成或存量实例引用的配置时禁止直接删除。
- 多人并发修改同一对象时提示冲突，保护已批准或已发布内容。
- 任务依赖形成闭环或引用无权项目时禁止保存。
- 必需检查项或交付件状态存在歧义时不得自动推进阶段。
- 已发布交付件不得被覆盖，修订必须创建新版本。
- 上游领域包升级与客户覆盖冲突时继续使用当前生产版本并要求裁定。
- 领域包迁移部分失败时，失败实例保持可恢复且不得形成不可解释混合状态。
- 自动化递归或重复触发时停止异常链路且不产生重复业务结果。
- 外部请求重复、乱序、过期或冲突时不得产生重复对象。
- 非关键外部系统不可用时待同步事项可追踪、重试和人工补偿。
- 权限撤销成功后，新在线请求立即使用最新权限；本地权限缓存最长 5 秒失效，活动会话最长 1 分钟撤销，搜索和报表投影最长 30 秒移除结果，但对象打开、导出和下载始终实时复核权限。
- 弱网同步冲突必须展示冲突字段，不得静默覆盖已审核内容。
- 报表数据非实时更新时必须显示数据截止时间。
- AI 或领域扩展不可用时核心项目流程仍可人工完成。
- 灾难恢复后审批、版本、附件引用、领域包版本和审计链保持一致。
- MySQL 的无符号整数、`TINYINT(1)`、零日期、字符集、排序规则、时区、`ENUM/SET`、JSON、BLOB 和自增键必须显式转换，禁止依赖隐式类型转换。
- 源库存在重复业务键、大小写冲突、尾随空格、孤儿外键或无主键表时必须隔离处理，不得静默合并或丢弃。
- 迁移切换后若 PDP 已产生新写入，不得假设可直接切回旧系统；必须执行经批准的反向同步或前向修复方案。
- 数据库产品对 NULL 排序、大小写、时区、JSON、锁、分页或事务隔离的默认行为不一致时，必须由平台统一语义或显式拒绝，不得向业务层泄露隐式差异。
- 已运行实例不得通过修改配置直接连接到另一空库或旧快照；数据库切换必须校验平台版本、schema 版本、数据位点和写入模式。

## 5. 需求（必需）

### 功能需求

#### 产品治理与工作空间（P1）

- **FR-001**: 本产品规格 MUST 作为所有 PDP 子规格的上位范围和优先级来源。
- **FR-002**: 子规格 MUST 声明所属层级、上位规格、引用需求和不在范围内容。
- **FR-003**: 平台 MUST 支持工作空间创建、启停、归档和负责人管理。
- **FR-004**: 工作空间 MUST 维护独立成员、角色、组织、配置、模板和数据范围。
- **FR-005**: 用户 MUST 能加入多个工作空间并在明确上下文中切换。
- **FR-006**: 跨工作空间共享 MUST 经显式授权并可撤销。
- **FR-121**: 每个项目 MUST 归属且仅归属一个主工作空间，由主工作空间承担项目编号、模板版本、数据主责、默认权限和审计责任。
- **FR-122**: 主工作空间 MAY 向其他工作空间授予项目、子项目或指定对象的协作权限，并明确范围、角色、期限、可执行动作和授权责任人。
- **FR-123**: 协作工作空间不得改变项目主工作空间、基础模板归属、数据保留责任或平台核心安全策略。
- **FR-124**: 跨工作空间授权的创建、变更、到期和撤销 MUST 在命令成功后立即用于新的在线请求并形成审计记录；本地权限缓存最长 5 秒失效，授权结束后历史链接不得继续访问。

#### 领域包与配置平台（P1）

- **FR-007**: 平台 MUST 支持领域包作为可版本、组合、发布、迁移和停用的业务能力单元。
- **FR-008**: 领域包 MUST 能扩展平台允许扩展的核心对象并创建全新业务对象。
- **FR-009**: 领域包 MUST 能定义字段、关系、唯一性、状态、动作、表单、页面、视图、规则、权限和集成映射。
- **FR-010**: 领域包不得覆盖核心对象身份、工作空间归属、基础权限、审计、版本和平台保留动作。
- **FR-011**: 领域定制 MUST 默认采用声明式配置；专业算法或设备工具 MAY 通过受治理、隔离的领域扩展接入。
- **FR-012**: 领域扩展不得修改平台核心、绕过权限审计或直接覆盖核心数据。
- **FR-013**: 领域包 MUST 采用平台标准包、行业领域包、工作空间客户包三层继承。
- **FR-014**: 每项覆盖 MUST 记录目标、原值、新值、原因、责任人和适用版本。
- **FR-015**: 领域包组合和升级 MUST 检测对象、字段、状态、规则、权限和集成冲突。
- **FR-016**: 领域设计与生产发布 MUST 强制职责分离。
- **FR-017**: 发布前 MUST 校验结构、引用、流程、权限、场景、迁移和回滚方案。
- **FR-018**: 运行中实例默认保持创建时的领域包版本快照。
- **FR-019**: 存量实例迁移 MUST 支持预览、映射、分批、报告、回滚和失败恢复。
- **FR-020**: 权限、合法性和状态约束 MUST 同步完成；通知、统计、搜索更新和非关键外部协同 MUST 可异步执行。
- **FR-130**: 平台 MUST 为工作空间、组织、成员、客户、合同、项目、阶段、任务、里程碑、交付件、审批、风险、问题和变更等公共对象定义统一核心字段、标识和业务语义。
- **FR-131**: 领域包 MAY 为允许扩展的公共对象增加扩展字段、关系、页面区域、视图、校验、规则和动作。
- **FR-132**: 领域包不得重复创建与平台核心字段语义相同的字段，不得改变核心字段的数据含义、唯一性、归属关系或平台级统计口径。
- **FR-133**: 跨领域搜索、组合报表、权限、审计和标准集成 MUST 优先使用统一核心字段；领域扩展字段 MUST 通过稳定标识和元数据参与这些能力。
- **FR-134**: 领域包发布前 MUST 检测扩展字段与核心字段的名称、语义、标识和数据来源冲突，并提供复用核心字段或明确差异的处理建议。

#### 项目模板、项目与阶段（P1）

- **FR-021**: 平台 MUST 支持从空白或模板创建项目。
- **FR-022**: 项目模板 MUST 包含默认阶段、任务、负责人规则、工期、里程碑、检查项、交付件、审批和视图。
- **FR-023**: 项目 MUST 维护编号、名称、目标、范围、客户、合同、负责人、成员、优先级、健康度和时间。
- **FR-024**: 项目 MUST 支持父子层级、区域执行和权限汇总。
- **FR-025**: 阶段 MUST 表达业务生命周期并配置进入条件、退出条件、负责人、计划和必需产出。
- **FR-026**: 阶段推进、回退、暂停、取消和关闭 MUST 受权限和完成条件控制。
- **FR-027**: 项目关闭 MUST 以必需任务、交付件、问题、审批和子项目达到条件为前提。
- **FR-028**: 项目复制、归档、恢复和作废不得删除历史和审计。
- **FR-117**: 平台 MUST 定义适用于所有项目的统一顶层生命周期，至少包含规划前、规划中、执行中、验收中、服务中、已关闭和已取消等管理状态。
- **FR-118**: 领域包 MAY 增加、拆分或命名领域子阶段，但每个可运行子阶段 MUST 映射到且仅映射到一个统一顶层生命周期状态。
- **FR-119**: 项目组合、跨项目报表、健康度和管理驾驶舱 MUST 使用统一顶层生命周期统计；项目工作区 MAY 同时展示领域子阶段。
- **FR-120**: 无法映射、映射冲突或会导致顶层生命周期倒退的领域阶段配置 MUST 在发布前被拒绝，受控回退场景除外。

#### 任务、计划与协作（P1）

- **FR-029**: 平台 MUST 支持任务和子任务的创建、分配、开始、阻塞、完成、取消和归档。
- **FR-030**: 任务 MUST 支持负责人、参与人、优先级、状态、计划与实际时间、工作量、评论、附件和关联对象。
- **FR-031**: 任务 MUST 支持强制和非强制检查项。
- **FR-032**: 强制检查项未完成时不得完成任务。
- **FR-033**: 平台 MUST 支持里程碑及其计划、实际、负责人和关联对象。
- **FR-034**: 平台 MUST 支持常用任务依赖并检测循环和日期冲突。
- **FR-035**: 平台 MUST 支持计划基线创建、批准、比较和历史。
- **FR-036**: 阶段、任务和里程碑 MUST 按统一可配置规则汇总项目进度。
- **FR-037**: 业务对象 MUST 支持评论、提及、关注、附件和活动时间线。
- **FR-125**: 项目统一进度 MUST 依据已批准计划中的里程碑权重及其必需任务、检查项和交付件完成状态计算，不得仅按任务数量占比计算。
- **FR-126**: 领域包 MAY 配置里程碑权重和完成规则，但同一项目的有效权重总和 MUST 为 100%，且每个权重项 MUST 具有明确业务成果和完成证据。
- **FR-127**: 未满足必需产出条件的里程碑不得计为完成；部分完成的计算规则 MUST 在领域包中明确并通过发布校验。
- **FR-128**: 项目进度 MUST 展示计算时间、适用基线、各里程碑贡献和阻塞项，使管理者可解释并追溯进度来源。
- **FR-129**: 人工调整项目进度 MAY 作为经审批的例外，但 MUST 记录原计算值、调整值、原因、期限和审批人，不得改变底层完成事实。

#### 交付件与审批（P1）

- **FR-038**: 交付件 MUST 作为独立对象管理类型、版本、状态、负责人、审批、签核和关联。
- **FR-039**: 已发布交付件不得被覆盖；修订 MUST 产生新版本。
- **FR-040**: 交付件 MUST 支持模板、命名规则、必需字段、到期和阶段约束。
- **FR-041**: 交付件 MUST 支持内部和外部签核并记录签核版本。
- **FR-042**: 项目归档 MUST 生成完整交付件清单并识别缺失和版本异常。
- **FR-043**: 平台 MUST 提供统一审批中心，支持所有声明可审批的对象。
- **FR-044**: 审批 MUST 支持通过、退回、拒绝、撤回、转交、委托、加签和抄送。
- **FR-045**: 审批 MUST 保留节点、办理人、时间、意见、附件、动作和状态回写。
- **FR-046**: 审批页面不得展示审批人无权查看的数据。

#### 风险、问题与变更（P1）

- **FR-047**: 风险 MUST 支持概率、影响、等级、负责人、措施、触发条件、日期和状态。
- **FR-048**: 风险 MUST 支持矩阵汇总和转化为问题。
- **FR-049**: 问题 MUST 支持来源、类型、严重程度、负责人、关联对象、方案和关闭结论。
- **FR-050**: 变更 MUST 记录范围、工期、资源、成本、风险和交付影响。
- **FR-051**: 已批准变更 MUST 更新关联对象并保留变更前后差异。
- **FR-052**: 风险、问题和变更重新打开、取消和关闭 MUST 记录原因。

#### 视图、搜索、通知与自动化（P1 基础体验，P2 自动化）

- **FR-053**: 平台 MUST 为项目和工作项提供概览、列表、看板、日历、时间线和详情视图。
- **FR-054**: 所有视图 MUST 使用一致数据、权限和状态。
- **FR-055**: 看板 MUST 支持状态映射、拖动更新、在制限制和阻塞标识。
- **FR-056**: 时间线 MUST 展示阶段、任务、里程碑、依赖、基线和延期。
- **FR-057**: 用户 MUST 能保存个人视图，管理员 MUST 能发布角色默认视图。
- **FR-058**: 平台 MUST 支持权限范围内的跨项目、跨对象搜索。
- **FR-059**: 统计卡片和图表 MUST 能钻取到有权明细。
- **FR-060**: P1 MUST 提供统一站内通知中心，支持核心任务、审批、交付件和风险事件的幂等通知、已读状态、对象链接、失败重提和权限复核。
- **FR-061**: P2 自动化规则 MUST 由事件、条件和动作组成并支持版本、范围和执行身份。
- **FR-062**: P2 自动化 MUST 防止重复与递归，失败不得破坏已完成核心事务。

#### 权限、安全、审计与数据（P1，FR-163 为 P2）

- **FR-063**: 平台 MUST 组合控制功能、数据范围、对象、字段和操作权限。
- **FR-064**: 数据范围 MUST 支持工作空间、组织、区域、客户、项目归属、参与关系和对象属性。
- **FR-065**: 临时授权 MUST 有范围、期限、审批和撤销能力。
- **FR-066**: 敏感字段、附件、签名、定位、设备凭据和成本 MUST 支持独立授权和审计。
- **FR-067**: 页面、搜索、报表、导出、通知、自动化、集成和缓存 MUST 使用一致权限。
- **FR-068**: 用户停用或权限撤销成功后，新的在线请求 MUST 立即使用最新授权；活动会话和刷新凭据最长 1 分钟撤销，未完成责任可重新分配。
- **FR-069**: 平台 MUST 记录业务、审批、配置、权限、登录、下载、导出、迁移和集成的关键审计事件。
- **FR-070**: 数据 MUST 支持分类、保留、归档、恢复、法律保留和处置规则。
- **FR-071**: 交付件、审批、签核和关键配置在保留期限前不得不可逆删除。
- **FR-163**: P2 的高级自动化、报表订阅和外部通知渠道 MUST 支持用户渠道偏好、版本化规则、执行身份、权限过滤、幂等、递归防护、失败重试和人工补偿。
- **FR-164**: 权限撤销后，搜索和报表投影 MUST 在 30 秒内移除无权结果；附件签名地址有效期不得超过 5 分钟且实际下载前必须复核权限；导出在启动、生成下载地址和下载时均必须复核权限；WebSocket 或 SSE 会话 MUST 在 30 秒内刷新授权或断开。

#### 项目组合、资源、成本与分析（P2）

- **FR-072**: 平台 MUST 支持项目组合及目标、健康度、进度、里程碑、风险、资源、成本和交付汇总。
- **FR-073**: 项目组合 MUST 支持按区域、客户、负责人、类型、状态、风险和时间筛选。
- **FR-074**: 平台 MUST 维护成员技能、区域、可用时间、产能和分配。
- **FR-075**: 平台 MUST 识别跨项目资源冲突并追溯至任务。
- **FR-076**: 任务 MUST 支持计划与实际工时。
- **FR-077**: 项目 MUST 支持预算、人工、采购、差旅、运输和外包等成本分类。
- **FR-078**: 资源或成本超阈值 MUST 产生预警但不得自动覆盖批准安排。
- **FR-079**: 平台 MUST 为主要角色提供默认驾驶舱。
- **FR-080**: 用户 MUST 能配置指标、图表、筛选、时间范围和布局。
- **FR-081**: 指标 MUST 定义口径、范围、周期、截止时间和明细来源。
- **FR-082**: 报表 MUST 支持查看、钻取、导出、订阅和定期分发。

#### 外部参与者、移动与集成（P2）

- **FR-083**: 平台 MUST 为内部人员、代理商、客户和现场工程师提供角色化入口。
- **FR-084**: 外部参与者只能访问明确授权的数据和动作。
- **FR-085**: 移动作业 MUST 支持任务、检查项、进度、照片、异常、签到和交付提交。
- **FR-086**: 弱网或短时离线 MUST 支持批准数据类型的安全暂存和可控同步。
- **FR-087**: 审批、阶段推进、基线变更和项目关闭 MUST 在可确认最新权限和数据状态时执行。
- **FR-088**: 平台 MUST 提供受授权的外部数据与业务操作能力。
- **FR-089**: 平台 MUST 提供关键业务事件订阅。
- **FR-090**: 外部请求 MUST 识别重复、乱序、过期和冲突。
- **FR-091**: 集成失败 MUST 支持记录、重试、暂停、取消和人工补偿。
- **FR-092**: 管理员 MUST 能查看集成健康度、成功时间、失败原因和积压。
- **FR-093**: 平台 MUST 支持客户、合同、采购、设备、审批、故障、备件和服务等标准集成场景。

#### MySQL 历史数据迁移（P1）

- **FR-139**: P1 上线 MUST 将现有 MySQL 历史数据迁移作为独立、可重复执行和可审计的上线工作流，不得通过人工直写平台生产核心表完成迁移。
- **FR-140**: 迁移前 MUST 盘点源库版本、表、视图、主键、约束、存储引擎、字符集、排序规则、时区、数据量、增长量、存储过程、触发器和数据质量。
- **FR-141**: 每个纳入迁移的源表和字段 MUST 具有目标对象、核心或扩展字段、转换规则、默认值、责任人和验收规则；不迁移内容 MUST 记录排除原因和处置方式。
- **FR-142**: 迁移 MUST 为新平台标识与 MySQL 原系统标识建立稳定映射，并保留源系统、源表、源主键、迁移批次和转换版本以供追溯。
- **FR-143**: 迁移 MUST 显式处理无符号整数、布尔语义、零日期、日期时区、精度、字符集与排序规则、`ENUM/SET`、JSON、BLOB、自增键、空值和空字符串等差异。
- **FR-144**: 当批准的停机窗口无法容纳全量迁移时，迁移 MUST 支持一致性全量快照后持续捕获增量变更，并在切换前证明增量已追平。
- **FR-145**: 迁移运行 MUST 支持预检、试运行、分批、断点续传、幂等重跑、失败隔离、进度、报告和受控回退。
- **FR-146**: 迁移核对 MUST 覆盖记录数量、关键业务汇总、标识映射、对象关系、状态、审批链、附件数量与内容哈希、必填字段和抽样业务场景。
- **FR-147**: 上线切换 MUST 具有写入冻结、最终增量、核对、启用 PDP、旧系统只读保留和明确 Go/No-Go 决策点。
- **FR-148**: 无效、冲突或无法映射的数据 MUST 进入隔离区并分级处置；迁移工具不得绕过平台权限、审计、字段合法性和状态约束。
- **FR-149**: 切换方案 MUST 定义回退点和不可逆点；PDP 开始产生新业务写入后，除非已验证反向同步，否则不得直接恢复旧系统为写入主系统。
- **FR-150**: 旧系统 MUST 在批准期限内只读保留，完成业务签字、审计证据归档和数据处置批准后方可下线。

#### 多数据库兼容与受控切换（P1）

- **FR-151**: P1 MUST 将 PostgreSQL 和 MySQL 作为首批同等级认证的关系数据库，任何一方不得成为业务功能、领域模型或外部契约的隐含前提。
- **FR-152**: 同一 PDP 版本制品 MUST 能通过部署配置选择已认证数据库，不得因数据库选择修改业务代码、领域包或重新构建应用。
- **FR-153**: 领域层和应用层 MUST 通过稳定持久化端口访问数据，不得直接依赖数据库专有 SQL、类型、函数、存储过程、触发器、序列、行级安全或全文检索语法。
- **FR-154**: 数据库差异 MUST 收敛在持久化适配器、方言、数据库变更集、查询实现和能力检测中；专有优化 MAY 使用，但 MUST 具有等价实现或可验证的降级路径。
- **FR-155**: 平台 MUST 定义跨数据库逻辑类型与行为基线，至少覆盖标识、布尔、文本、定点数、日期时间、JSON、二进制、NULL、排序规则、唯一性、分页、锁和事务隔离。
- **FR-156**: 数据库 schema 变更 MUST 具有公共变更集和按数据库隔离的专用变更集，并支持校验和、重复执行保护、升级前检查和失败恢复；禁止人工修改生产 schema 作为正式发布步骤。
- **FR-157**: 搜索、报表、权限过滤、审计、事件日志和领域扩展查询 MUST 在所有认证数据库上保持相同业务语义；任何数据库原生优化不得成为正确性的唯一来源。
- **FR-158**: CI 和发布门禁 MUST 在每种认证数据库上执行完整的持久化契约、数据库变更、模块集成、权限、迁移和核心端到端测试。
- **FR-159**: 应用启动和升级前 MUST 校验数据库产品、版本、字符集、排序规则、时区、事务存储引擎、必要扩展和 schema 版本，并对不兼容环境快速失败。
- **FR-160**: 生产环境每个 PDP 部署 MUST 在任一时刻只有一个写入主数据库；不支持在运行中仅修改连接配置完成无迁移热切换，也不得形成未经治理的跨数据库双主写入。
- **FR-161**: 已认证数据库之间的切换 MUST 使用版本化、可重放的标准迁移包或等价受控通道，覆盖全量、增量、标识映射、核对、冻结、切换、回退点和审计证据。
- **FR-162**: 每种认证数据库 MUST 具有达到相同可用性、RTO、RPO、备份、时间点恢复、故障切换和容量目标的部署与运行手册。

#### 行业领域包（P1 标准实施，其他行业包 P2）

- **FR-094**: P1 标准实施领域包 MUST 覆盖工前准备、施工计划、实施方案、部署、验收和归档主线。
- **FR-095**: P2 售前测试领域包 MUST 支持申请、审批、测试跟踪、回访和闭环。
- **FR-096**: P2 转包代施领域包 MUST 支持申请、供应方、审批、合同执行、付款、交付和回访。
- **FR-097**: P2 割接领域包 MUST 支持分级、方案、审批、操作步骤、验证、回退和闭环。
- **FR-098**: P2 验收回访领域包 MUST 支持培训、满意度、签核、交付件和闭环。
- **FR-099**: P2 维保领域包 MUST 支持维护计划、服务记录、问题、交付件和客户资产关联。
- **FR-100**: P2 六类行业包 MUST 能扩展核心对象或创建新对象，同时共享权限、审批、审计、搜索、报表和集成。
- **FR-101**: P2 客户差异 MUST 通过工作空间客户包显式覆盖，不得复制后断开上游关系。
- **FR-135**: P1 首个可上线版本 MUST 同时交付平台基础能力和标准实施领域包，不得仅以配置设计器或平台管理后台作为可上线产品。
- **FR-136**: P1 标准实施领域包闭环 MUST 至少覆盖项目创建、工前准备、施工计划、实施方案、实施部署、交付件、审批、验收和项目归档。
- **FR-137**: P1 上线验收 MUST 使用真实或等价试点项目完成端到端业务验证，并验证平台配置、权限、视图、审计和高可用要求。
- **FR-138**: P2 售前测试、转包代施、割接、回访和维保领域包 MAY 在 P1 期间进行设计或原型验证，但不得作为 P1 首次上线的必需交付条件。

#### 高可用、恢复与性能（P1）

- **FR-102**: 核心项目、任务、审批和交付操作月度可用性 MUST ≥ 99.95%。
- **FR-103**: 其他在线管理能力月度可用性 MUST ≥ 99.9%。
- **FR-165**: 月度可用性 MUST 按“成功完成且未超过适用交互时限的合格请求数 ÷ 合格请求总数”计算；客户端主动取消以及客户控制的网络或身份系统故障可排除，PDP 应用、数据库、对象存储和内部依赖故障不得排除，割接保障窗口和未批准维护不得从核心可用性中排除。
- **FR-104**: 割接保障窗口核心功能 MUST 无计划停机，故障后 5 分钟内通知值守人员。
- **FR-105**: 核心业务恢复时间目标 MUST ≤ 30 分钟，数据恢复点目标 MUST ≤ 5 分钟。
- **FR-106**: 报表、搜索、AI、领域扩展或非关键集成故障时核心项目操作 MUST 可继续或明确降级。
- **FR-107**: 关键数据 MUST 持续保护并至少每季度验证恢复。
- **FR-108**: 平台 MUST 支持 1000 名并发活跃用户。
- **FR-109**: 目标数据规模下 95% 常用页面操作 MUST ≤ 2 秒呈现可用结果。
- **FR-110**: 百万级记录下 95% 常用搜索和报表 MUST ≤ 3 秒返回或先呈现可交互渐进结果。
- **FR-111**: 大批量导入、导出、归档和统计 MUST 后台处理并展示进度和失败明细。

#### 跨域治理、运行与体验（P1）

- **FR-166**: 每个 P1 业务能力 MUST 明确参与角色、触发与输入、状态变化、业务规则、输出、异常处理、补偿路径、验收责任人和闭环证据位置。
- **FR-167**: 工作空间、领域包、项目、阶段、任务、里程碑、基线、交付件、审批、风险、问题、变更、后台作业和迁移 MUST 使用确定性状态机；每个迁移 MUST 定义前置条件、权限、并发语义、结果和稳定失败原因。
- **FR-168**: 领域包发布与迁移、项目或阶段回退、基线替换、人工进度调整、交付件发布、审批终态、数据导出与处置、历史迁移和数据库切换等高风险操作 MUST 提供影响预览、明确确认、审计以及撤销、回退或人工补偿路径。
- **FR-169**: 每项 P1 关键能力 MUST 具有服务等级档案，至少定义服务类别、适用交互时限、SLI/SLO、容量假设、失败模式、告警条件、责任人、运行手册和证据位置。
- **FR-170**: P1 核心旅程 MUST 支持键盘操作、清晰焦点、可识别控件名称以及不依赖颜色的状态和错误表达，并通过自动化和代表性用户验证。
- **FR-171**: P1 设计和发布 MUST 完成威胁建模、数据分类、传输与存储保护、密钥和凭据管理、职责分离、跨空间隔离及审计防篡改评审；未批准的严重或高风险项不得进入实现或发布。
- **FR-172**: P1 外部 HTTP、事件和领域包接口 MUST 具有完整版本化契约、消费者与兼容影响记录、弃用窗口和自动化兼容校验；契约变更及失败测试必须早于提供方或消费者实现。

#### 国际化、高级无障碍与 AI（P3，FR-113 为 P1 核心基线）

- **FR-112**: 平台 MUST 支持语言、时区、日期、时间和数字格式本地化。
- **FR-113**: P1 核心页面 MUST 支持键盘操作、清晰焦点和不依赖颜色的状态表达；P3 扩展屏幕阅读器深度适配和更广泛的无障碍能力。
- **FR-114**: AI MAY 提供摘要、行动项、风险建议和授权知识问答。
- **FR-115**: AI 建议未经用户确认不得改变业务数据、触发审批或向外发送信息。
- **FR-116**: AI 不可用、建议被拒绝或用户未确认时，同一核心业务动作 MUST 保留人工入口，且 AI 不得持有业务状态变更或审批触发的唯一控制权。

### 关键实体

- **Workspace**：组织、配置、权限和业务数据的隔离协作边界。
- **DomainPackage**：对象、页面、流程、规则、权限和集成的版本化业务能力包。
- **ProjectTemplate**：项目阶段、任务、里程碑、检查项、交付件和审批的复用方案。
- **Project / Stage**：项目及其生命周期区段。
- **Task / ChecklistItem**：可分配执行单元及完成条件。
- **Milestone / Dependency / PlanBaseline**：计划节点、约束和批准快照。
- **Deliverable**：可版本、审批、签核、发布和归档的成果。
- **Approval**：独立于业务对象的授权决策过程。
- **Risk / Issue / ChangeRequest**：风险、已发生问题和受控变更。
- **SavedView / InAppNotification**：P1 个人或角色视图配置，以及幂等、可追踪的站内通知。
- **RetentionPolicy / LegalHold / DispositionRequest**：P1 数据保留、法律保留和批准处置对象。
- **OperationalAlert / RecoveryOperation / AvailabilityDailySummary**：P1 告警、恢复和可用性证据对象。
- **ResourceProfile / Worklog / CostRecord**：P2 资源、投入和成本信息。
- **Portfolio / Dashboard / Report**：P2 跨项目管理和分析对象。
- **Workflow / AutomationRule**：P2 高级流程自动化规则和执行对象。
- **IntegrationConnection**：P2 外部授权、数据范围和运行状态。
- **AuditEvent**：不可由普通用户修改的关键行为记录。
- **MigrationProgram / MigrationRun / LegacyKeyMap**：历史数据迁移范围、运行批次、问题、核对结果及新旧标识映射。
- **DatabaseDeploymentProfile**：当前认证数据库类型、版本、schema、能力探测和运行约束的部署事实。
- **UserAccount / UserSession**：企业身份映射、停用状态、会话与刷新凭据生命周期、语言和时区偏好的平台账户。

### 服务等级、性能和验收口径

#### P1 服务目录

| 服务类别 | 典型请求 | 适用交互时限 | 可用性与失败处理 |
|---|---|---|---|
| 核心交付命令 | 项目/阶段推进、任务更新、审批办理、交付件提交与发布 | 95% 在 2 秒内返回明确成功、拒绝或冲突结果 | 纳入 99.95%；不得依赖搜索、报表或非关键外部系统 |
| 核心交付查询 | 项目概览、任务与待办、审批和交付件详情 | 95% 在 2 秒内呈现可操作结果 | 纳入 99.95%；缓存失效时回源或明确降级 |
| 其他在线管理 | 配置设计、审计检索、运维控制台、迁移控制台 | 95% 在 3 秒内呈现结果或后台作业入口 | 纳入 99.9%；失败必须可重试或人工补偿 |
| 搜索和基础统计 | 权限搜索、项目基础统计和跨视图汇总 | 95% 在 3 秒内返回，或 1 秒内呈现可交互渐进结果与数据截止时间 | 纳入 99.9%；不可用时核心流程保留人工导航 |
| 后台作业 | 导入、导出、归档、统计、投影重建和迁移 | 5 秒内返回作业标识；后续持续展示进度、检查点和失败明细 | 不以同步请求时长衡量，按作业接受率、完成率和恢复能力衡量 |

可用性统计中的核心与其他在线请求 MUST 使用版本化服务目录维护；每项请求明确服务类别、时限、
责任团队和允许排除原因。客户端主动取消及客户控制的网络或身份系统故障可排除，平台应用、数据库、
对象存储、内部依赖、未批准维护和割接保障窗口故障不得排除。

#### 性能与容量基准

- 性能数据集至少包含 10 个工作空间、10,000 个项目、1,000,000 个任务、100,000 个审批实例、
  200,000 个交付件版本、2,000,000 条活动/审计记录及能够产生权限过滤和跨空间协作的数据分布。
- 事务集至少覆盖项目工作区打开、任务列表过滤、任务状态更新、审批打开与办理、交付件列表与发布、
  权限搜索、基础统计、导出提交以及迁移/数据库切换控制台查询。
- 在生产等价拓扑和认证数据库上，预热 10 分钟后持续测量不少于 30 分钟；并发用户按浏览、查询和
  写入混合模型运行，失败率不得超过 0.1%，并分别报告 P50、P95、P99、吞吐、资源和连接池等待。
- PostgreSQL 与 MySQL 使用相同数据生成规则、事务集和验收阈值；任何单库专有优化必须同时提供
  另一认证数据库的等价路径或通过已批准的降级验收。

#### 真实或等价试点最低标准

“真实或等价试点项目”至少包含 1 个主工作空间、3 类角色、1 个主项目和 2 个子项目、6 个业务阶段、
30 个任务、5 个里程碑、10 个交付件版本、3 条审批链、5 条风险/问题/变更记录、2 个并发或失败场景，
并包含一批可追溯历史数据。试点必须从创建运行到归档，覆盖权限、多视图、状态机、审计、降级和恢复；
删减任一最低项必须由产品、领域和质量负责人书面批准。

#### 高风险操作目录

| 操作 | 必需预览与确认 | 撤销、回退或补偿 |
|---|---|---|
| 领域包发布、升级和实例迁移 | 版本差异、受影响对象、冲突、不可逆变更和迁移批次 | 保持原生产版本、分批暂停、回滚或失败实例隔离 |
| 项目/阶段受控回退和项目关闭 | 未完成事项、影响交付件、审批、子项目和统计口径 | 取消操作或按受控回退恢复前一合法状态 |
| 基线替换和人工进度调整 | 原值、新值、原因、期限、贡献变化和审批人 | 恢复系统计算值或重新启用前一基线 |
| 交付件发布和审批终态 | 内容哈希、版本、签核、对象状态和下游影响 | 创建修订、撤回允许的审批或执行明确业务补偿 |
| 数据导出、法律保留解除和处置 | 数据范围、敏感级别、数量、保留约束和不可逆点 | 取消作业、撤销下载地址、停止处置或按批准恢复 |
| 历史迁移和认证数据库切换 | 映射、数量、关系、增量位点、差异、冻结窗口和单写主权 | 回退点前恢复源系统；不可逆点后执行已验证前向修复或反向同步 |

## 7. 假设

- PDP 首要服务于企业内部项目交付，并允许受控邀请代理商和客户参与。
- Workspace 是逻辑隔离和配置边界，不等同于商业计费租户。
- 项目阶段、任务状态和审批状态保持分离。
- 平台核心对象稳定，行业差异优先通过领域包表达。
- 领域包采用声明式配置为主、受治理扩展为辅的模型。
- 领域包固定采用平台标准、行业领域、工作空间客户三层继承。
- 外部系统仍对其主数据和内部流程负责。
- 资源管理首期不包含自动智能排程，成本管理不替代正式会计核算。
- 移动端聚焦现场采集、任务、待办和签核。
- AI 为辅助能力，不是核心流程依赖。
- P1 只交付基础权限搜索和站内通知；高级自动化、报表订阅和外部通知渠道属于 P2。
- 每个 PDP 部署在同一时刻使用一个已认证关系数据库作为业务事实真源；多数据库支持不等于单实例跨数据库分片或在线双写。
- 具体法规、认证、保留期限和首批语言在相应子规格或计划中确认。

## 8. 依赖

- 企业身份和组织数据源。
- 客户、合同、采购、设备、审批、故障、备件和消息渠道等外部系统协作条件。
- 项目治理方提供产品优先级、数据分类、指标口径和领域包发布责任人。
- 领域专家提供标准实施、售前、转包、割接、验收和维保业务样例。
- 试点团队提供真实数据、验收场景和上线前量化基线。
- 现有 MySQL 系统负责人提供只读访问、结构说明、业务口径、变更窗口、binlog 或等价增量来源以及数据签字责任人。
- 部署与运维团队提供 PostgreSQL 或 MySQL 的认证版本、备份恢复、故障切换、监控、容量和升级能力。

## 9. 风险

- 产品范围较大，若未按 P1/P2/P3 和规格层级推进，可能重新演变为无边界需求池。
- 配置能力过度复杂会提高管理成本，需要模板、权限、测试和发布治理。
- 行业业务过早进入平台内核会削弱升级能力和复用性。
- 既有数据质量和主数据责任不清会影响迁移、搜索和报表可信度。
- 灵活权限模型可能难以理解，需要权限预览和访问解释。
- 离线写入、双向同步、领域包迁移和扩展运行属于高风险能力，需分阶段验证。
- MySQL 历史数据可能包含类型、字符集、状态、标识、附件和业务语义不一致；若未提前盘点和彩排，会直接扩大上线停机和回退风险。
- 追求数据库可移植性可能导致复杂查询能力退化或形成两套实现；必须以公共业务语义、适配器边界和双数据库测试矩阵控制分叉。
- 跨数据库切换若缺少一致性快照、增量追平和明确写入主权，可能形成数据分叉；禁止把修改连接字符串当作迁移方案。
