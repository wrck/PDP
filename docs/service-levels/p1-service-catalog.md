**特性目录**：`specs/002-pdp-product`

**创建日期**：2026-07-17

**状态**：P1 实现基线（版本化目录）

**规格层级**：运行能力子规格，自顶向下追溯至 L0 产品规格 `specs/002-pdp-product/spec.md`

**主要追溯**：FR-102～FR-111、FR-162、FR-164、FR-165、FR-174；SC-016、SC-018、SC-037；宪章原则 VIII（可观测、可恢复与安全迁移）

**主要干系人**：运维/SRE、架构负责人、各业务模块负责人、数据迁移负责人、安全/合规负责人、业务签字人

**维护责任**：`operations` 模块负责人统筹目录版本与月度报告；各能力类别由对应责任模块共同维护并签字。

## 1. 目的与范围

本目录是 PDP 项目 P1 阶段关键能力的服务等级档案（Service Level Profile）权威目录，定义每类关键能力的请求分类、交互时限、SLI/SLO、容量假设、失败模式、告警条件、责任人和运行手册。

本目录满足宪章原则 VIII 中“每项关键能力 MUST 定义 SLI/SLO、容量假设、失败模式、告警条件、关联标识和运行手册”的不可协商要求，并支撑 SC-037 月度可审计报告中“服务目录覆盖率、请求分类覆盖率、交互时限覆盖率和排除项证据完整率均为 100%”的验收口径。

**覆盖范围**：P1 全部关键在线能力与受治理运行能力，每类关键能力 100% 纳入档案，无孤立页面或接口被视作已交付能力。P2/P3 能力不进入本目录；新增 P2/P3 能力必须先建立独立子规格或 backlog，获批后重新执行 clarify、plan、tasks、analyze，再纳入对应版本目录。

**包含的能力类别**：

1. 核心交付命令与查询
2. 配置、审计和运维管理
3. 搜索和基础统计
4. 后台作业
5. 历史迁移与上线切换
6. 平台工作流编排
7. 权限与授权决策
8. 附件与对象存储访问

**版本治理**：本目录以语义化版本维护，新增能力类别或调整 SLO 阈值为 MINOR，澄清、错字或证据路径修订为 PATCH；任何调整必须同步更新 `specs/002-pdp-product/plan.md` 的“P1 服务等级与容量矩阵”及 `checklists/requirements.md`，并保持月度报告追溯一致。

## 2. SLI/SLO 定义规范

### 2.1 可用性计算公式

按 FR-165 与 SC-016 统一口径，月度可用性按以下公式计算：

```
月度可用性 = 成功完成且未超过适用交互时限的合格请求数 ÷ 合格请求总数
```

等价表述为基于可用时间的统计：

```
月度可用性 = 月度可用时间 ÷ (月度总时间 − 批准排除项时长)
```

两式在统一请求分类、交互时限和排除项后必须等价，并以请求成功率口径为准。

**SLO 阈值**：

- 核心业务（核心交付命令与查询、平台工作流编排中影响核心事务的子能力、权限决策对核心事务的实时路径）：月度可用性 ≥ 99.95%（FR-102、SC-016）。
- 其他在线能力（配置、审计、运维管理、搜索与基础统计、附件访问等）：月度可用性 ≥ 99.9%（FR-103、SC-016）。
- 历史迁移与上线切换不按上述在线可用性统计，而按 RTO/RPO 与单写主权口径度量（见 §3.5）。
- 后台作业不按同步请求时长衡量，按作业接受率、完成率与检查点可恢复率度量（见 §3.4）。

### 2.2 交互时限

- 核心交付命令与查询：P95 ≤ 2 秒（FR-109、SC-018）。
- 配置、审计和运维管理：P95 ≤ 3 秒。
- 搜索和基础统计：P95 ≤ 3 秒，或在 1 秒内呈现可交互渐进结果并附带数据截止时间（FR-110、SC-018）。
- 后台作业：5 秒内接受请求并返回作业标识（FR-111）。
- 平台工作流编排：流程定义部署、实例启动、人工任务查询等在线管理操作 P95 ≤ 3 秒；异步执行器接受任务 P95 ≤ 1 秒。
- 权限与授权决策：权限校验作为核心事务的一部分必须在核心交互时限（P95 ≤ 2 秒）内完成；权限撤销按 §3.7 时限生效。
- 附件与对象存储访问：签名地址生成 P95 ≤ 2 秒；下载首字节时间受对象存储 SLA 约束，但平台侧鉴权复核必须在核心交互时限内完成。

所有交互时限以 P95 为基线测量，并在容量测试中同时记录 P99 用于异常诊断与告警阈值设定；SLO 报告以 P95 为准。

### 2.3 容量单位

- 在线交互能力：以“活跃并发用户数”与“规格事务混合”为单位（FR-108，1000 活跃并发）。
- 数据规模：以规格数据集对象数量为单位（10 工作空间、10,000 项目、1,000,000 任务、100,000 审批实例、200,000 交付件版本、2,000,000 活动/审计记录）。
- 后台作业：以“批次规模”与“资源预算（线程池/连接池/队列容量）”为单位。
- 历史迁移：以“全量数据量 + 增量位点延迟”为单位。
- 平台工作流：以“流程定义数 + 在途实例数 + 异步任务数”为单位。

### 2.4 失败率口径

- 请求失败率 = 失败请求数 ÷ 合格请求总数。SC-018 要求容量测试窗口内失败率 ≤ 0.1%。
- “失败请求”指：返回 5xx、超时（超过适用交互时限且未降级）、返回 4xx 中因平台内部错误导致的结果；不含客户端主动取消、客户控制的网络或身份系统故障导致的失败（这些进入排除项）。
- 4xx 中因业务规则拒绝（如权限不足、版本冲突、前置条件不满足）不计入失败率，但纳入业务正确性统计。

### 2.5 报告与证据

每月由 `operations` 模块按 SC-037 生成可审计报告，覆盖：服务目录覆盖率、请求分类覆盖率、交互时限覆盖率、排除项证据完整率，四项均须达到 100%。报告进入 `specs/002-pdp-product/evidence/` 与运维可用性报告库，保留至法规与保留策略要求期限。

## 3. 服务等级档案目录

### 3.1 核心交付命令与查询

承载项目/阶段推进、任务更新、审批办理、交付件提交与发布等核心写命令，以及项目概览、任务与待办、审批和交付件详情等核心查询。是 PDP 业务闭环的主干，纳入 99.95% 可用性口径。

| 字段 | 内容 |
|---|---|
| 请求分类 | 核心交付命令（项目/阶段推进、任务更新、审批办理、交付件提交与发布）；核心交付查询（项目概览、任务与待办、审批和交付件详情） |
| 交互时限 | 命令 P95 ≤ 2 秒、P99 ≤ 4 秒；查询 P95 ≤ 2 秒、P99 ≤ 4 秒 |
| SLI 名称 | `pdp.core.command.availability`、`pdp.core.query.availability`、`pdp.core.latency.p95`、`pdp.core.latency.p99` |
| SLO 阈值 | 月度可用性 ≥ 99.95%；P95 ≤ 2 秒；容量测试失败率 ≤ 0.1% |
| 容量假设 | 1000 活跃并发用户，按规格事务混合（项目工作区打开、任务列表过滤、任务状态更新、审批打开与办理、交付件列表与发布）；数据规模按规格数据集 |
| 主要失败模式 | 应用进程故障（OOM/线程池耗尽/异常未捕获）；MySQL 主库故障或连接池耗尽；对象存储不可用影响交付件发布；权限校验失败或事件登记失败导致事务回滚；乐观锁冲突未正确返回 409 |
| 告警条件 | 5 分钟内通知值守人员（FR-104）；可用性低于 99.95% 滚动 1 小时窗口触发 P1 告警；P95 延迟超过 2 秒持续 5 分钟触发 P2 告警；事务失败率超过 0.1% 触发 P2 告警；MySQL 主库连接池使用率超过 80% 触发 P2 告警 |
| 责任人 | `operations` 与对应业务模块（`project`、`planning`、`approval`、`deliverable`）共同负责；割接保障窗口由运维/SRE 值守 |
| 运行手册路径 | `docs/runbooks/core-delivery.md`（命令）、`docs/runbooks/core-query.md`（查询） |
| 最近验证证据路径 | `specs/002-pdp-product/evidence/availability/core-delivery-<YYYY-MM>.md` |

**降级原则**：核心事务不得依赖搜索、报表、AI 或非关键外部系统（FR-106）；Redis 缓存失效时核心事务降级为直接访问数据库；查询缓存失效时回源或明确降级。

承载权限过滤的跨项目/跨对象基础搜索、项目基础统计和跨视图汇总。纳入 99.9% 可用性口径，但不可用时核心流程必须保留人工导航能力。

| 字段 | 内容 |
|---|---|
| 请求分类 | 权限搜索；项目基础统计；跨视图汇总；保存视图查询 |
| 交互时限 | P95 ≤ 3 秒，或在 1 秒内呈现可交互渐进结果并附带数据截止时间；P99 ≤ 6 秒 |
| SLI 名称 | `pdp.search.availability`、`pdp.search.latency.p95`、`pdp.search.projection.lag`、`pdp.search.revocation.propagation` |
| SLO 阈值 | 月度可用性 ≥ 99.9%；P95 ≤ 3 秒或 1 秒渐进结果；投影位点滞后 ≤ 30 秒；权限撤销投影移除 ≤ 30 秒（FR-164） |
| 容量假设 | 1,000,000 个任务、2,000,000 条活动/审计记录；数据分布覆盖权限过滤与跨空间协作场景 |
| 主要失败模式 | 搜索投影滞后超过 30 秒；投影重建期间性能下降；权限撤销延迟导致无权结果残留（违反 FR-164）；全文检索加速失效回退到数据库扫描；保存视图权限范围变更未同步 |
| 告警条件 | 投影位点滞后超过 30 秒触发 P2 告警并在界面显示数据截止时间；投影重建进度低于阈值触发 P2 告警；权限撤销后 30 秒内投影未移除无权结果触发 P1 告警；搜索 P95 超过 3 秒持续 5 分钟触发降级并显示截止时间 |
| 责任人 | `experience` 与 `operations` 共同负责；权限撤销传播由 `identity` 与 `experience` 联合验证 |
| 运行手册路径 | `docs/runbooks/search-projection.md`、`docs/runbooks/statistics-degradation.md` |
| 最近验证证据路径 | `specs/002-pdp-product/evidence/availability/search-<YYYY-MM>.md`；证据必须包含投影位点与撤权时限验证记录 |

**降级原则**：搜索与基础统计不可用时核心交付流程保留人工导航（FR-106）；搜索词项由平台统一分析器生成，数据库全文检索只作为候选加速，不能直接决定权限、最终结果集合或稳定排序。

### 3.4 后台作业

承载导入、导出、归档、统计、投影重建和迁移批次等长耗时任务。不以同步请求时长衡量，按作业接受率、完成率与检查点可恢复率度量。

| 字段 | 内容 |
|---|---|
| 请求分类 | 批量导入；批量导出；归档处置；统计汇总；搜索投影重建；迁移批次 |
| 交互时限 | 接受请求并返回作业标识 P95 ≤ 5 秒（FR-111）；后续持续展示进度、检查点和失败明细 |
| SLI 名称 | `pdp.job.accept.latency.p95`、`pdp.job.completion.rate`、`pdp.job.checkpoint.recovery.rate`、`pdp.job.deadletter.count` |
| SLO 阈值 | 5 秒内接受并返回作业标识；作业完成率 ≥ 99%（按月度统计，排除经审批的人工取消）；检查点可恢复率 100% |
| 容量假设 | 导入、导出、归档、统计和迁移批次受资源预算约束（独立线程池、队列容量、连接预算）；与在线业务共享模块和数据库但不争用核心事务资源 |
| 主要失败模式 | 队列积压导致接受延迟超过 5 秒；线程池耗尽；部分失败（个别记录失败但批次继续）；死信（重试耗尽形成 incident/dead-letter）；检查点丢失导致无法恢复 |
| 告警条件 | 队列积压超过阈值触发 P2 告警；线程池使用率超过 80% 触发 P2 告警；部分失败率超过批次阈值触发 P2 告警并暂停批次；死信数量超过 0 触发 P1 告警并要求人工补偿；检查点写入失败触发 P1 告警 |
| 责任人 | 作业所属业务模块（`planning`、`deliverable`、`governance`、`experience`、`datamigration` 等）与 `operations` 共同负责 |
| 运行手册路径 | `docs/runbooks/background-jobs.md`、`docs/runbooks/job-deadletter.md` |
| 最近验证证据路径 | `specs/002-pdp-product/evidence/availability/jobs-<YYYY-MM>.md`；证据包含进度、失败明细与补偿记录 |

**重放与补偿**：死信支持安全重放与人工补偿，不得重复形成审批结论或业务状态变化；批量导入、导出、归档和统计统一使用可暂停、可恢复、可取消的后台作业。

### 3.5 历史迁移与上线切换

承载 MySQL 旧系统到 PDP MySQL 的全量迁移、增量追平、业务核对、冻结切换与旧系统退役。不按在线可用性统计，按 RTO/RPO 与单写主权口径度量。

| 字段 | 内容 |
|---|---|
| 请求分类 | 全量迁移批次；增量位点捕获与追平；业务核对；冻结-切换-启用单向流程；旧系统只读保留与退役 |
| 交互时限 | 切换窗口内 RTO ≤ 30 分钟（FR-105）；RPO ≤ 5 分钟（FR-105）；切换为单向不可逆点 |
| SLI 名称 | `pdp.migration.rto`、`pdp.migration.rpo`、`pdp.migration.single.writer`、`pdp.migration.reconciliation.gap` |
| SLO 阈值 | RTO ≤ 30 分钟；RPO ≤ 5 分钟；单写主权 100%（同一时刻只有一个业务写入主库）；核对差异为 0 方可切换 |
| 容量假设 | 两次全量彩排 + MySQL 旧系统到 PDP 的单向切换；源数据规模按规格数据集与历史 MySQL 盘点结果 |
| 主要失败模式 | 增量位点落后超过 RPO；业务核对失败（数量/状态/权限/审计差异）；误启用双写违反单写主权；回退点越界（已过不可逆点）；源系统 schema 差异导致映射失败 |
| 告警条件 | 增量落后超过阈值立即阻断切换并告警；核对差异非 0 立即阻断切换；检测到双写立即阻断并告警；回退点越界立即阻断并升级至业务签字人；任何阻断状态必须在告警中明确不可逆点位置 |
| 责任人 | `datamigration`、`persistence`、DBA 与业务签字人共同负责；切换决策必须业务签字人书面批准 |
| 运行手册路径 | `docs/runbooks/mysql-migration.md`、`docs/runbooks/cutover-switch.md`、`docs/runbooks/cutover-rollback.md` |
| 最近验证证据路径 | `specs/002-pdp-product/evidence/migration/rehearsal-<n>.md`、`specs/002-pdp-product/evidence/migration/cutover-<YYYY-MM-DD>.md` |

**单写主权与不可逆点**：不采用长期双写；PDP 开始接收新写入后，回切旧系统必须先有经过验证的反向同步，否则采用前向修复；切换采用“冻结旧系统写入—应用最终增量—核对—启用 PDP—旧系统只读”的单向流程，并明确回退点与不可逆点。

### 3.6 平台工作流编排

承载 Flowable Process Engine 的 BPMN 定义部署、流程实例、人工任务、定时器/消息、异步执行器、重试/补偿和流程版本迁移。是审批、项目交付和领域包复用的平台公共基础能力。

| 字段 | 内容 |
|---|---|
| 请求分类 | 流程定义部署与版本管理；流程实例启动与推进；人工任务查询与办理；定时器/消息关联；异步执行器任务接受；流程版本迁移预览与执行；incident/dead-letter 检索与重放 |
| 交互时限 | 在线管理操作 P95 ≤ 3 秒、P99 ≤ 6 秒；异步执行器接受任务 P95 ≤ 1 秒；流程实例启动作为核心事务一部分时纳入核心交互时限 |
| SLI 名称 | `pdp.workflow.definition.deploy.availability`、`pdp.workflow.instance.latency.p95`、`pdp.workflow.task.query.latency.p95`、`pdp.workflow.async.executor.backlog`、`pdp.workflow.incident.count` |
| SLO 阈值 | 影响核心事务的子能力月度可用性 ≥ 99.95%；纯编排管理操作月度可用性 ≥ 99.9%；异步执行器积压不超过阈值；incident 数量为 0（出现即触发告警） |
| 容量假设 | 流程定义数与在途实例数按规格数据集（100,000 审批实例）；异步执行器使用独立线程池、队列、连接预算；Flowable 表与 PDP 业务表使用独立 schema/账号/HikariCP 池 |
| 主要失败模式 | 流程引擎故障（启动失败/DDL 不一致）；异步执行器积压导致任务延迟；重试耗尽形成 incident/dead-letter；流程版本迁移预览失败；引擎事件桥接中断导致 outbox 与流程状态不一致 |
| 告警条件 | 引擎启动或 DDL 校验失败立即阻断应用启动并告警；异步积压超过阈值触发 P2 告警；incident 数量超过 0 触发 P1 告警并要求人工补偿；流程版本迁移预览失败触发 P2 告警；引擎事件桥接滞后超过阈值触发 P2 告警 |
| 责任人 | `workflow` 与 `operations` 共同负责；引擎升级边界由架构负责人签字 |
| 运行手册路径 | `docs/runbooks/workflow-engine.md`、`docs/runbooks/workflow-async-executor.md`、`docs/runbooks/workflow-incident.md`、`docs/runbooks/workflow-version-migration.md` |
| 最近验证证据路径 | `specs/002-pdp-product/evidence/availability/workflow-<YYYY-MM>.md` |

**编排边界**：Flowable 只负责编排协调，领域状态机与审批聚合仍是业务事实源；流程实例和人工任务不得成为项目、任务、交付件、审批、权限或审计事实的唯一存储；任何重试不得生成重复审批动作或业务结果。

### 3.7 权限与授权决策

承载页面、搜索、报表、导出、附件、通知、自动化、集成和缓存的统一权限决策与撤销传播。是核心事务正确性的前提，撤销时限为强约束。

| 字段 | 内容 |
|---|---|
| 请求分类 | 在线请求权限校验；权限缓存查询与失效；搜索/报表投影权限移除；会话授权刷新与断开；导出/下载权限复核 |

## 5. 可用性排除项

按 FR-165 与 SC-037 口径，可用性计算中的排除项必须可审计、可追溯，并具有责任方和证据。排除项证据完整率必须达到 100%。

### 5.1 可排除项

- **批准维护窗口**：经产品负责人与运维/SRE 书面批准的计划维护窗口，且在维护前已公告并避开割接保障窗口。排除需记录批准人、公告时间、窗口起止、影响范围。
- **外部依赖故障且核心降级运行**：客户控制的网络或身份系统（OIDC IdP）故障导致的核心访问失败，但 PDP 应用本身仍可降级运行（如核心事务通过本地缓存或备用鉴权继续）。排除需记录故障依赖、降级路径、降级持续时间。
- **不可抗力**：自然灾害、数据中心级故障等不可抗力导致的停机。排除需记录事件性质、影响范围、恢复措施。

### 5.2 不可排除项

以下故障不得从核心可用性中排除（FR-165）：

- PDP 应用自身故障（进程崩溃、OOM、配置错误、部署失败）。
- MySQL 主库故障（包括主从切换、连接池耗尽、锁等待超时）。
- 对象存储故障（影响交付件发布与下载）。
- 内部依赖故障（Redis、Flowable 引擎、内部事务事件日志）。
- 未批准维护（任何未走批准流程的停机）。
- 割接保障窗口内的计划停机（割接保障窗口核心功能 MUST 无计划停机，FR-104）。

### 5.3 排除项审批与证据

每个排除项必须包含：责任方、批准人、生效日期、失效日期（如适用）、故障证据（告警、日志、复盘报告）、影响范围与持续时间、复盘改进项。排除项按月归档至 `specs/002-pdp-product/evidence/availability/exclusions-<YYYY-MM>.md`，并在月度报告中列明。

## 6. 容量测试方法

容量测试遵循 SC-018 与 plan.md 的口径，用于冻结 SLO 阈值与容量假设。

### 6.1 数据集

按规格定义的性能数据集，至少包含：

- 10 个工作空间
- 10,000 个项目
- 1,000,000 个任务
- 100,000 个审批实例
- 200,000 个交付件版本
- 2,000,000 条活动/审计记录
- 能够产生权限过滤和跨空间协作的数据分布

### 6.2 事务集

事务集至少覆盖：项目工作区打开、任务列表过滤、任务状态更新、审批打开与办理、交付件列表与发布、权限搜索、基础统计、导出提交以及迁移/数据库切换控制台查询。

### 6.3 测试拓扑

在生产等价拓扑下执行：三副本部署、独立 HikariCP 池、Flowable 独立 schema/账号/数据源、对象存储与 Redis 容器等价配置。测试环境与生产环境的差异必须记录并评估对结果的影响。

### 6.4 测试流程

1. **预热**：在生产等价拓扑下预热 10 分钟，使缓存、连接池、JIT、Flowable 引擎达到稳态。
2. **持续测量**：持续 30 分钟测量窗口，1000 并发混合负载按规格事务混合比例施压。
3. **采样**：每 15 秒采样一次延迟分布、吞吐量、失败率、资源使用率（CPU、内存、连接池、线程池、队列）。
4. **验证**：核心交互 P95 ≤ 2 秒，搜索和基础统计 P95 ≤ 3 秒或 1 秒内呈现渐进结果，失败率 ≤ 0.1%（SC-018）。
5. **冻结**：容量测试通过后冻结 SLO 阈值与容量假设，作为后续告警阈值与容量规划的基线。

### 6.5 专项测试

- **权限穿透测试**：验证撤权时限（新请求立即/缓存 5 秒/投影 30 秒/会话 1 分钟）与越权阻断率。
- **审计覆盖检查**：验证关键审计事件防篡改与可检索。
- **数据库恢复演练**：每季度验证关键数据恢复（FR-107），含时间点恢复与故障切换。
- **迁移彩排**：至少两次 MySQL 生产等价全量迁移与彩排切换（见 §3.5）。

## 7. 复核条件

本目录及各档案在以下条件触发时必须复核，复核必须记录继续有效、修订或例外结论及责任人。

### 7.1 容量测试后冻结阈值

每次容量测试完成后，根据测试结果冻结或修订 SLO 阈值与容量假设。修订必须更新版本号、同步 plan.md 的“P1 服务等级与容量矩阵”及 `checklists/requirements.md`。

### 7.2 SLO 连续未达标

任一能力连续两个月度未达 SLO 阈值，或单月度偏差超过 0.05%（核心） / 0.1%（其他），必须触发专项复核：评估阈值合理性、容量假设、失败模式覆盖与告警有效性，并形成改进项进入可追踪计划。不得通过改变基线或统计口径宣称达标。

### 7.3 新增关键能力

新增 P1 关键能力时，必须先在本目录建立对应档案（含全部 §4 字段）并冻结运行手册与证据路径，方可进入实现。新增 P2/P3 能力不进入本目录，需先建立独立子规格或 backlog。

### 7.4 季度复核

至少每季度复核一次本目录，覆盖：SLO 达成情况、排除项证据完整性、运行手册有效性、告警准确率、容量假设与实际负载偏差、失败模式覆盖完整性。季度复核由 `operations` 模块负责人组织，各能力责任模块共同签字。

### 7.5 事件驱动复核

发生重大生产事故、监管变化、产品战略调整、认证数据库变化或架构边界变化时，必须触发专项复核，并完成无责复盘，改进项进入可追踪计划（宪章原则 VIII）。
