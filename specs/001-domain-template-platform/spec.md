# Feature Specification: PDP 可扩展领域模板平台

**Feature Directory**: `specs/001-domain-template-platform`

**Created**: 2026-07-16

**Status**: Draft

**Input**: 在 PDP 项目中建设企业级项目交付平台。`docs/000-init` 下的业务规格、调研和建议仅作为参考材料；本 feature 聚焦平台内核与领域模板的深度可扩展能力，确保平台配置后仍能细化标准实施、售前测试、转包代施、割接、验收、维保等复杂业务。

## Product Context

PDP 采用“稳定平台内核 + 可版本化领域模板”的产品模型：

```text
Platform
└── Workspace
    ├── Domain Package
    │   ├── Business Object Type
    │   ├── Field / Relation / Validation
    │   ├── State Model / Workflow / Approval
    │   ├── Form / Page / View / Dashboard
    │   ├── Rule / Action / Notification
    │   └── Integration Contract
    └── Project
        ├── Stage
        ├── Task / Checklist
        ├── Deliverable
        └── Domain Object Instance
```

平台内核提供身份、权限、工作空间、项目、任务、交付件、审批、审计、搜索、通知、规则执行和开放集成等稳定能力。领域模板通过领域包定义业务对象、关系、状态、页面、规则、动作和集成映射，在不修改平台核心语义的前提下表达具体业务。

## Clarifications

### Session 2026-07-16

- Q: 领域深度定制采用哪种扩展模型？ → A: 声明式配置为主，允许受治理、隔离运行的领域扩展；领域扩展不得修改或绕过平台核心。
- Q: 领域包如何使用平台核心对象？ → A: 既可扩展核心对象，也可创建新领域对象，但核心对象的身份、归属、权限、审计和基础语义不可覆盖。
- Q: 领域规则和扩展采用何种执行一致性边界？ → A: 核心校验与状态约束同步执行；通知、统计、搜索更新和外部协同等后续动作异步执行。
- Q: 领域包采用何种继承与客户定制模型？ → A: 固定采用平台标准包→行业领域包→工作空间客户包三层继承，并允许显式、可追踪的覆盖。
- Q: 领域包生产发布采用何种治理方式？ → A: 领域设计者提交候选版本，由独立发布者审核测试、影响范围和迁移方案后发布。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建领域包与业务对象 (Priority: P1)

领域管理员从空白或标准领域包派生业务模板，定义业务对象、字段、对象关系、唯一性约束和生命周期，使平台能够承载项目基础对象之外的行业数据。

**Why this priority**：只有支持扩展业务对象，领域模板才能表达割接单、工勘记录、设备、维保记录等复杂业务，而不是停留在修改项目字段。

**Independent Test**：创建“割接管理”领域包，新增割接单、割接步骤和割接资源三类对象，建立项目、设备、交付件之间的关系，发布后在测试工作空间创建和查询实例。

**Acceptance Scenarios**：

1. **Given** 管理员具有领域设计权限，**When** 创建业务对象并配置字段、关系和唯一性规则，**Then** 系统能够校验、发布并创建该对象实例
2. **Given** 新对象引用项目和设备，**When** 查看任一关联对象，**Then** 用户可在权限范围内查看双向关联

---

### User Story 2 - 配置状态、流程与业务动作 (Priority: P1)

领域管理员为业务对象定义状态、允许动作、转换条件、审批、自动操作和异常分支，使不同业务能够拥有独立且可测试的生命周期。

**Why this priority**：复杂领域不仅需要表单，还需要可表达审批、回退、取消、补偿和并行活动的行为模型。

**Independent Test**：为割接单定义草稿、评审、待执行、执行中、验证、闭环和取消状态，配置审批与回退条件，验证合法和非法流转。

**Acceptance Scenarios**：

1. **Given** 对象未满足转换条件，**When** 用户执行状态动作，**Then** 系统阻止转换并列出缺失条件
2. **Given** 审批被退回，**When** 发起人修订后重新提交，**Then** 对象回到适用状态并保留完整历史

---

### User Story 3 - 定制表单、页面与视图 (Priority: P1)

领域管理员按业务对象、状态和角色配置录入表单、详情页面、操作区、列表、看板、日历、时间线和统计视图，使页面结构能够适配领域业务，而不是只有固定项目页面。

**Why this priority**：字段可配置但页面不可配置，仍会导致复杂业务依赖定制开发。

**Independent Test**：为工勘记录配置分步表单、条件显示、设备明细子表和只读审核页，为项目经理和审批人配置不同页面。

**Acceptance Scenarios**：

1. **Given** 字段显示条件不满足，**When** 用户填写表单，**Then** 无关字段隐藏且不参与必填校验
2. **Given** 审批人与录入人角色不同，**When** 打开同一对象，**Then** 两者看到各自授权的布局、字段和动作

---

### User Story 4 - 配置复杂校验、计算与规则 (Priority: P1)

领域管理员定义跨字段、跨明细、跨对象和跨阶段的校验、计算、完成条件与自动化规则，用于表达工期倒推、物料符合性、交付件齐套和主子项目联动等业务。

**Why this priority**：复杂业务规则不能只靠字段必填和简单状态流表达。

**Independent Test**：配置“主项目仅在全部子项目闭环后可关闭”和“割接前必须完成指定交付件与检查项”规则，验证触发、提示和审计结果。

**Acceptance Scenarios**：

1. **Given** 跨对象规则依赖的数据发生变化，**When** 用户执行受规则约束的动作，**Then** 系统使用最新有权数据重新判断
2. **Given** 规则执行失败，**When** 用户保存或转换状态，**Then** 系统明确区分业务不满足与规则执行异常

---

### User Story 5 - 领域包组合、继承与覆盖 (Priority: P1)

工作空间管理员将多个兼容领域包组合为业务方案，并在受控范围内覆盖字段、页面、流程、规则和权限，支持“平台标准 + 行业标准 + 客户差异”的分层定制。

**Why this priority**：只有继承和组合机制，才能避免为每个客户复制整套模板并长期分叉。

**Independent Test**：组合“标准项目交付”“网络设备”“客户 A 合规要求”三个领域包，验证冲突检测、覆盖优先级和最终生效配置。

**Acceptance Scenarios**：

1. **Given** 两个领域包修改同一状态或字段规则，**When** 管理员组合发布，**Then** 系统展示冲突并要求明确裁定
2. **Given** 上游标准包发布兼容更新，**When** 工作空间评估升级，**Then** 系统展示本地覆盖与上游变化的差异

---

### User Story 6 - 版本发布、迁移与回滚 (Priority: P1)

领域管理员以草稿、测试、发布、停用和回滚方式管理领域包版本，并选择新项目使用新版本或将存量项目受控迁移。

**Why this priority**：领域配置会持续演进，必须避免发布后破坏运行中的项目。

**Independent Test**：从 v1 增加新字段和状态生成 v2，分别验证新项目使用 v2、存量项目保持 v1、兼容迁移和失败回滚。

**Acceptance Scenarios**：

1. **Given** 领域包仍存在无效引用或不兼容变更，**When** 管理员发布，**Then** 系统阻止发布并列出影响范围
2. **Given** 存量实例迁移中部分失败，**When** 迁移任务结束，**Then** 成功和失败对象分别可见，失败对象保持可恢复状态

---

### User Story 7 - 权限、审计与安全边界 (Priority: P1)

平台和领域管理员分别管理平台能力与领域配置权限；领域对象沿用统一的数据范围、字段权限和审计规则，领域模板不得绕过平台安全边界。

**Why this priority**：越灵活的配置能力越需要明确治理，否则领域管理员可能无意扩大数据权限。

**Independent Test**：配置代理商可编辑现场记录但不可查看成本和其他代理商对象，验证页面、搜索、报表、导出、自动化和集成都遵循相同权限。

**Acceptance Scenarios**：

1. **Given** 领域规则尝试读取无权字段，**When** 规则执行，**Then** 系统拒绝并记录安全审计事件
2. **Given** 管理员修改对象、流程或权限配置，**When** 审计员查询，**Then** 可查看版本、操作者、时间、前后值和发布结果

---

### User Story 8 - 领域扩展与外部协同 (Priority: P2)

当声明式配置不足以表达专业算法、设备工具或外部业务时，平台通过受治理的扩展点调用领域能力，并统一管理授权、版本、失败隔离和运行状态。

**Why this priority**：平台必须允许深度扩展，同时保护核心稳定性和升级能力。

**Independent Test**：为设备配置解析注册一个领域扩展，模拟正常、超时、重复和失败结果，验证核心事务、重试和审计不受破坏。

**Acceptance Scenarios**：

1. **Given** 领域扩展不可用，**When** 用户执行相关业务动作，**Then** 平台按配置选择阻止、降级或转人工，并保持核心数据一致
2. **Given** 扩展版本升级，**When** 新版本不兼容既有领域包，**Then** 系统阻止启用并显示依赖关系

### Edge Cases

- 业务对象或字段标识与平台保留标识冲突时，禁止保存。
- 对象关系形成不允许的循环级联时，禁止发布并展示关系路径。
- 删除被页面、规则、报表、集成或存量数据引用的配置时，禁止直接删除。
- 状态模型存在不可达状态、无终态或无限自动循环时，禁止发布。
- 规则之间形成递归触发时，停止异常链路并记录完整执行路径。
- 多领域包定义同名对象但语义不同时，不得自动合并。
- 上游模板升级与本地覆盖冲突时，保持当前生产版本并要求人工裁定。
- 字段类型变更导致存量值不可转换时，不得自动迁移。
- 迁移失败时，不得产生同一对象一部分使用新模型、一部分使用旧模型的不可解释状态。
- 领域扩展超时或失败时，不得长期占用或破坏平台核心业务事务。
- 领域页面隐藏字段不得绕过服务端校验和权限。
- 自动化、报表、搜索、导出和集成必须使用与页面一致的领域权限。
- 工作空间复制领域包时，不得复制密钥、成员、客户数据或历史实例。

## Requirements *(mandatory)*

### Functional Requirements

#### 领域元模型

- **FR-001**: 平台 MUST 支持领域包作为可独立版本、发布、组合和停用的配置单元。
- **FR-002**: 领域包 MUST 能定义业务对象类型、字段、对象关系、唯一性约束、状态模型、动作、页面、视图、规则、权限和集成映射。
- **FR-003**: 业务对象 MUST 支持单值字段、多值字段、明细集合、成员、组织、附件、富文本、计算值和其他对象引用。
- **FR-004**: 对象关系 MUST 支持一对一、一对多、多对多、父子聚合和普通引用，并明确删除、归档和权限传播行为。
- **FR-005**: 每个对象、字段、状态、动作和关系 MUST 具有在领域包范围内稳定且唯一的标识。
- **FR-006**: 平台 MUST 保留项目、任务、交付件、审批、成员、组织和审计等核心对象语义，领域包不得修改其不可变安全约束。
- **FR-007**: 领域对象 MUST 能关联平台核心对象，并参与搜索、权限、活动记录、通知、报表和审计。
- **FR-061**: 领域包 MUST 能为项目、任务、交付件、风险、问题和变更等允许扩展的核心对象增加领域字段、关联对象、页面区域、视图、校验、规则和业务动作。
- **FR-062**: 领域包 MUST 能创建具有独立字段、关系、状态、页面、权限和行为的全新领域对象。
- **FR-063**: 核心对象允许扩展的范围 MUST 由平台显式声明；领域包不得覆盖核心对象标识、工作空间归属、基础访问控制、审计要求、版本控制和平台保留动作。
- **FR-064**: 核心对象扩展与新领域对象 MUST 使用统一的查询、权限、活动记录、通知、报表、自动化和集成能力，不得形成独立且不可治理的数据孤岛。

#### 状态、流程与动作

- **FR-008**: 每个可流转对象 MUST 能定义状态、初始状态、终态、允许动作和状态转换。
- **FR-009**: 状态转换 MUST 支持角色、字段、关系、检查项、交付件、审批结果和时间条件。
- **FR-010**: 业务动作 MUST 能定义适用对象、可见条件、执行条件、输入表单、确认提示和结果。
- **FR-011**: 流程 MUST 支持通过、退回、拒绝、撤回、取消、暂停、恢复和受控回退。
- **FR-012**: 领域流程 MUST 能调用平台统一审批，但不得将审批状态与业务生命周期状态视为同一状态。
- **FR-013**: 发布前 MUST 检测不可达状态、循环自动转换、缺少终态、缺少责任人和无效引用。
- **FR-014**: 所有状态和动作变化 MUST 形成不可由普通业务用户修改的历史记录。

#### 页面、表单与视图

- **FR-015**: 领域包 MUST 能为对象配置创建、编辑、查看、审核和外部参与者表单。
- **FR-016**: 表单 MUST 支持分组、页签、分步、明细表、关联对象选择和条件显示。
- **FR-017**: 字段 MUST 支持显示、隐藏、必填、只读、默认值和校验规则，并能按对象状态和角色变化。
- **FR-018**: 领域对象 MUST 能配置列表、看板、日历、时间线、统计和关联对象视图。
- **FR-019**: 管理员 MUST 能配置详情页的区域、字段、关联列表、活动记录和动作位置。
- **FR-020**: 页面配置不得扩大用户对对象、字段或动作的实际权限。
- **FR-021**: 用户 MUST 能保存个人筛选和布局；工作空间管理员 MUST 能发布角色默认视图。

#### 校验、计算与规则

- **FR-022**: 平台 MUST 支持字段级、对象级、集合级和跨对象业务校验。
- **FR-023**: 规则 MUST 能读取当前对象、关联对象、父子项目、任务、交付件和审批结果中的有权数据。
- **FR-024**: 平台 MUST 支持计算字段、聚合指标、编号规则、完成条件和风险判定。
- **FR-025**: 自动化规则 MUST 由事件、条件和动作组成，并明确作用范围、执行身份、优先级和版本。
- **FR-026**: 规则执行 MUST 防止重复和递归，失败 MUST 可追踪、可重试且不破坏已完成的核心操作。
- **FR-027**: 用户触发的业务校验失败 MUST 返回可理解的业务原因；平台执行异常 MUST 与业务不满足明确区分。
- **FR-028**: 规则或计算使用的数据发生变化后，平台 MUST 按配置重新计算或标记结果过期。
- **FR-065**: 权限判断、字段与对象合法性、唯一性、状态转换条件、强制完成条件和核心数据一致性规则 MUST 在业务操作返回成功前同步完成。
- **FR-066**: 通知发送、统计汇总、搜索更新、非关键文档生成和外部系统同步 MUST 作为可追踪的异步后续动作执行，除非领域包明确声明该外部结果属于核心业务成功条件。
- **FR-067**: 异步后续动作失败 MUST 不得回滚已经合法完成的核心业务事务；平台 MUST 记录失败、展示影响并支持重试、补偿或转人工。
- **FR-068**: 领域包声明某扩展属于同步核心条件时，MUST 明确超时、失败、降级和人工替代策略，并在发布前通过故障场景校验。

#### 组合、继承与覆盖

- **FR-029**: 领域包 MUST 采用固定三层继承模型：平台标准包、行业领域包、工作空间客户包；不得增加第四层或形成任意深度继承。
- **FR-030**: 行业领域包 MUST 从平台标准包派生，工作空间客户包 MUST 从适用行业领域包派生；派生包 MUST 记录继承来源、本地新增、本地覆盖和被禁用配置。
- **FR-031**: 工作空间 MUST 能组合多个兼容领域包，并生成最终生效模型。
- **FR-032**: 组合时 MUST 检测对象、字段、状态、动作、页面、规则、权限和集成契约冲突。
- **FR-033**: 冲突不得静默覆盖；管理员 MUST 明确选择保留、覆盖、映射或取消组合。
- **FR-034**: 上游领域包升级时，平台 MUST 展示新增、删除、修改及与本地覆盖的差异。
- **FR-035**: 工作空间本地覆盖 MUST 限于领域包声明允许覆盖的范围。
- **FR-036**: 平台核心安全、审计和数据隔离规则 MUST 不允许被领域包覆盖。
- **FR-069**: 每项行业或客户覆盖 MUST 标识覆盖目标、原值、新值、原因、责任人和适用版本，不得以复制后断开关系的方式隐藏继承来源。
- **FR-070**: 上游包升级时，平台 MUST 对下游覆盖重新执行兼容性与冲突评估；未完成评估前，下游生产环境继续使用当前已发布组合版本。

#### 版本、测试与迁移

- **FR-037**: 领域包 MUST 支持草稿、测试、已发布、已停用和已归档状态。
- **FR-038**: 每次发布 MUST 生成不可变版本，并记录说明、操作者、时间和依赖。
- **FR-039**: 新项目或对象实例 MUST 绑定明确的领域包版本快照。
- **FR-040**: 新版本默认不得自动改变运行中实例。
- **FR-041**: 平台 MUST 提供领域包测试环境，用于验证页面、权限、状态、规则、自动化和迁移。
- **FR-042**: 发布前 MUST 运行结构校验、引用校验、流程校验、权限校验和样例场景测试。
- **FR-043**: 存量实例迁移 MUST 提供影响预览、数据映射、兼容性检查、分批执行、结果报告和回滚方案。
- **FR-044**: 删除字段、状态或关系时 MUST 明确存量数据的保留、转换、归档或禁止策略。
- **FR-045**: 迁移过程 MUST 保持对象、关系、审批、附件和审计引用的一致性。

#### 权限与治理

- **FR-046**: 平台 MUST 区分平台管理员、工作空间管理员、领域设计者、领域发布者和业务使用者权限。
- **FR-047**: 领域设计与生产发布 MUST 强制职责分离；同一用户不得同时作为某一候选版本的主要设计者和最终发布者。
- **FR-048**: 领域对象 MUST 继承平台统一的工作空间、组织、项目关系、对象和字段权限能力。
- **FR-049**: 领域规则、自动化、报表、搜索、导出和集成 MUST 使用明确执行身份并遵循相同权限。
- **FR-050**: 领域配置、发布、回滚、迁移、权限和扩展运行 MUST 记录审计事件。
- **FR-051**: 平台 MUST 提供领域配置依赖图和引用查询，使管理员能够评估变更影响。
- **FR-052**: 领域包 MUST 声明所有必要权限、数据访问范围、外部依赖和敏感字段。
- **FR-071**: 领域设计者 MUST 能创建和测试候选版本并提交发布申请，但不得自行将其提交的候选版本发布至生产。
- **FR-072**: 领域发布者在批准生产发布前 MUST 审核校验结果、场景测试、权限影响、依赖变化、存量实例影响、迁移方案和回滚方案。
- **FR-073**: 发布申请被退回后，设计者 MUST 能修订并重新提交；每轮审核意见、修订和决定 MUST 完整保留。
- **FR-074**: 紧急回滚 MAY 由具备紧急发布权限的独立发布者执行，但 MUST 记录原因、影响和事后复核结果。

#### 扩展与集成

- **FR-053**: 领域包 MUST 能声明业务事件、外部数据映射、调用动作和失败处理策略。
- **FR-054**: 外部协同 MUST 支持授权范围、版本、重复识别、重试、暂停、补偿和运行状态查询。
- **FR-055**: 声明式规则无法满足的领域能力 MUST 通过受治理扩展点接入，不得直接修改平台核心对象的不可变约束。
- **FR-056**: 领域扩展 MUST 声明适用领域包版本、输入、输出、权限、超时、失败和兼容性信息。
- **FR-057**: 领域扩展失败 MUST 与平台核心事务隔离，并按领域配置选择阻止、降级、排队或转人工。
- **FR-058**: 扩展升级前 MUST 检查所有依赖领域包和运行中实例的兼容性。
- **FR-059**: 领域定制 MUST 默认使用声明式对象、页面、流程、规则与集成配置；仅当声明式能力无法合理表达专业算法、设备工具或特殊外部业务时，才允许注册领域扩展。
- **FR-060**: 领域扩展 MUST 在受治理的隔离边界内运行，不得修改平台核心定义、绕过统一权限与审计、直接覆盖核心数据，或使平台升级依赖客户私有核心改动。

### Key Entities

- **DomainPackage**：可版本、组合、发布和迁移的领域配置集合。
- **DomainPackageVersion**：领域包的不可变发布快照。
- **BusinessObjectType**：领域业务对象类型定义。
- **FieldDefinition**：字段语义、数据形态、校验和权限定义。
- **RelationDefinition**：对象之间关系及生命周期行为定义。
- **StateModel**：对象状态、动作、转换和条件定义。
- **PageDefinition**：对象表单、详情、审核和外部页面定义。
- **ViewDefinition**：列表、看板、日历、时间线和统计视图定义。
- **RuleDefinition**：校验、计算、完成条件或自动化规则。
- **DomainExtension**：声明式配置之外的受治理领域能力。
- **PackageComposition**：多个领域包在工作空间中的组合及冲突裁定结果。
- **MigrationDefinition**：版本间数据和状态迁移规则。
- **DomainObjectInstance**：依据领域包版本创建的业务数据实例。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 领域管理员可在 60 分钟内创建并发布一个包含 3 类关联对象、20 个字段、1 套状态模型、3 个页面和5 条规则的可用领域包。
- **SC-002**: 标准实施、售前测试、转包代施、割接、验收和维保六类参考业务均能通过领域包表达其专有对象、关系、流程、页面和规则。
- **SC-003**: 领域包结构错误、无效引用、不可达状态、循环自动化和权限越界配置的发布前识别率 100%。
- **SC-004**: 新领域包或新版本发布不会自动改变运行中实例，非授权配置漂移率为 0。
- **SC-005**: 兼容升级的存量实例迁移成功率 ≥ 99.5%，失败实例 100% 可定位且保持可恢复状态。
- **SC-006**: 页面、搜索、报表、导出、自动化和集成对领域对象的数据权限一致率 100%。
- **SC-007**: 领域配置、发布、覆盖、迁移和扩展运行的审计覆盖率 100%。
- **SC-008**: 上游领域包升级时，本地覆盖冲突和潜在破坏性变更识别率 100%。
- **SC-009**: 领域扩展故障时，平台核心项目、任务、交付件和审批能力不受破坏，故障隔离率 100%。
- **SC-010**: 80% 的领域需求可由受培训的领域管理员通过配置完成，无需修改平台核心产品。
- **SC-011**: 所有领域扩展均具备明确版本、权限、依赖和失败策略，直接修改平台核心或绕过平台安全边界的扩展数量为 0。
- **SC-012**: 六类参考业务均能同时使用核心对象扩展和新领域对象表达业务，且核心对象不可变语义被覆盖的情况为 0。
- **SC-013**: 权限、合法性和状态约束在操作成功返回前完成判断的覆盖率 100%；异步后续动作失败导致已完成核心事务被错误回滚的情况为 0。
- **SC-014**: 所有生产领域包均可追溯至平台、行业和工作空间三层来源，超过三层继承或无法识别来源的配置数量为 0。
- **SC-015**: 生产领域包版本经过独立发布审核的比例 100%，设计者自行发布其候选版本的情况为 0。

## Assumptions

- `docs/000-init` 中的文件仅作为业务和产品研究输入，不是 Spec Kit 活动 feature。
- 平台核心对象保持稳定，复杂行业差异优先通过领域对象和领域包表达。
- 领域模板不是简单项目模板；本规格统一使用“领域包”表示可包含对象、页面、流程、规则和集成的完整业务能力包。
- 工作空间可以组合多个领域包，但生产发布必须经过校验和授权。
- 具体技术实现、脚本语言、运行时隔离方式和扩展部署模式在计划阶段确定。
- 领域包详细内容将在标准实施、售前测试、转包代施、割接、验收和维保等后续子 feature 中分别定义。
