# PDP 高风险操作目录与统一交互规范

## 1. 目的

本文件定义 PDP 高风险操作目录，并统一预览、确认、不可逆点和补偿的交互规范。本文件同时预注册
认证数据库切换操作类型、禁用原因和 P2 启用条件，但在 P1 不提供执行入口。

本规范服务于宪章原则 IX「结果导向与可解释的用户体验」与 `data-model.md` §2.1
「状态迁移与高风险操作通用规则」：所有高风险界面必须展示对象版本、影响范围、阻断项、不可逆点
和补偿路径；确认前不得执行写入。

## 2. 高风险操作定义

高风险操作是指操作影响范围大、不可逆或难以补偿的写操作。所有高风险操作 MUST 统一接入：

- 影响预览（`OperationImpactPreview`）；
- 明确的用户确认；
- 审计记录；
- 撤销 / 回退 / 补偿框架。

通用失败原因至少包含：无权、前置条件未满足、非法状态、版本冲突、依赖冲突、保留 / 法律约束、
迁移门禁未通过和不可逆点已越过。失败响应和审计必须包含 `correlation_id`、目标对象、当前状态、
当前版本、稳定原因和下一步建议，但不得泄露无权字段。

## 3. OperationImpactPreview 结构

依据 `data-model.md` §2 公共值对象，`OperationImpactPreview` 字段如下：

| 字段 | 含义 |
|---|---|
| `operation_key` | 高风险操作稳定键，对应目录中的操作键 |
| `target_refs` | 操作目标对象引用列表（`ObjectRef`） |
| `current_revisions` | 生成预览时各目标对象的当前版本 |
| `impacts` | 影响范围描述（受影响对象、范围、统计口径变化等） |
| `blocking_items` | 阻断项（不解决不得执行的操作前置条件） |
| `irreversible_point` | 不可逆点说明（越过该点后只能前向修复） |
| `compensation_options` | 可用补偿路径（撤销 / 回退 / 补偿） |
| `expires_at` | 预览过期时间 |

**执行约束**：高风险操作执行时必须引用未过期预览，且对象版本未发生变化。版本变化时必须重新生成
预览，不得静默覆盖。

## 4. 统一交互规范

### 4.1 预览阶段

- 展示对象版本（`current_revisions`）、影响范围（`impacts`）、阻断项（`blocking_items`）、
  不可逆点（`irreversible_point`）和补偿路径（`compensation_options`）。
- 预览必须有过期时间（`expires_at`），过期后不得用于执行。
- 对象版本变化时必须重新生成预览，不得静默覆盖；旧预览必须失效。
- 预览生成必须经过权限校验，无权对象不得出现在 `target_refs` 中。

### 4.2 确认阶段

- 用户明确确认后才能执行写入；确认需引用预览版本（`operation_key` + 目标对象 + `current_revisions`）。
- 高风险写操作必须使用幂等键，避免重复执行造成叠加影响。
- 确认动作必须记录确认人、确认时间、引用的预览标识和确认时上下文。

### 4.3 执行阶段

- 执行记录必须保存：
  - 预览摘要（影响范围、阻断项、不可逆点、补偿选项）；
  - 确认人；
  - 实际影响（与预览差异，如有）；
  - 补偿结果；
  - 不可逆点是否已越过及越过时间。
- 执行必须以 `expected_revision` 进行；对象版本变化时拒绝执行并提示重新生成预览。
- 执行失败必须返回稳定失败原因和 `correlation_id`，并保留可诊断证据。

### 4.4 补偿阶段

- 高风险操作必须提供撤销 / 回退 / 补偿路径，并在预览中明示可用路径。
- 不可逆点（`irreversible_point`）越过后只能前向修复，不得回滚至不可逆点之前。
- 补偿动作本身若属于高风险操作，必须同样接入预览 / 确认 / 审计 / 补偿框架。
- 已发布配置、审批、签核、已发布交付件在保留期内不可物理删除，补偿只能通过受控处置或前向修复完成。

## 6. 认证数据库切换预注册

依据 `plan.md` 阶段 A0 要求，P1 部署仅支持单一认证 MySQL 8.4 业务事实真源，认证数据库跨库切换
属于 P2 范围。本操作在 P1 仅预注册，不提供执行入口。

### 6.1 预注册内容

- **操作类型**：`auth_db.switch`，登记于本目录。
- **状态**：P1 禁用执行，仅返回稳定禁用原因；P2 启用前必须通过切换治理。
- **权限**：P1 不授予执行权限；P2 启用条件达成后通过独立治理流程授予。
- **能力探测扩展点**：持久化适配器能力画像必须能声明「是否支持跨库认证切换」，扩展接口预留但
  不连接或认证第二种生产数据库。

### 6.2 P1 稳定禁用原因

P1 调用本操作时必须返回稳定禁用原因（不依赖于具体部署细节）：

> 认证数据库跨库切换属于 P2，当前部署仅支持单一认证数据库。

禁用原因必须纳入通用失败原因分类（前置条件未满足 / 迁移门禁未通过），并可通过审计和契约测试验证。

### 6.3 P2 启用条件

满足以下全部条件后，方可启动 P2 启用治理流程：

1. 第二种目标数据库认证完成（持久化适配器能力画像声明支持）。
2. 跨库迁移执行器通过验证（迁移、回退、写入冻结、切换、回退点能力）。
3. 切换治理通过（含本目录预览 / 确认 / 审计 / 补偿框架接入、回退演练和保留约束验证）。

P2 启用前必须更新本目录条目、权限矩阵和契约测试，并完成端到端抽查覆盖率 100% 验证。

## 7. 通用失败原因

高风险操作失败响应必须使用稳定原因分类，不得使用非确定文案：

- 无权（不泄露无权字段）；
- 前置条件未满足；
- 预览过期；
- 版本冲突（对象版本变化）；
- 不可逆点已越过；
- 补偿失败；
- 非法状态；
- 依赖冲突；
- 保留 / 法律约束；
- 迁移门禁未通过。

失败响应必须包含 `correlation_id`、目标对象、当前状态、当前版本、稳定原因和下一步建议。

## 8. 验证要求

高风险操作必须通过以下测试，端到端抽查覆盖率 100%：

- **影响预览过期测试**：预览过期后执行必须被拒绝，并提示重新生成预览。
- **并发版本变化测试**：预览生成后对象版本变化时，执行必须以 `expected_revision` 拒绝并提示
  重新生成预览，不得静默覆盖。
- **确认测试**：未明确确认不得执行写入；幂等键重复请求必须返回同一结果。
- **补偿通用测试**：不可逆点前可补偿；不可逆点后只能前向修复；补偿失败必须留下可诊断证据。
- **认证数据库切换预注册测试**：P1 调用 `auth_db.switch` 必须返回稳定禁用原因，不执行实际切换；
  能力探测扩展点必须声明不支持跨库认证切换。
- **测试载体**：单元 / 契约层 `HighRiskOperationTest`；端到端层 `e2e/high-risk-operation.spec.ts`。

## 9. 复核条件

满足以下任一条件时，必须对本目录进行复核：

- 新增高风险操作。
- 补偿路径变更。
- P2 数据库切换启用前。
- 季度例行复核，确认操作目录、预览结构、失败原因和测试覆盖与现行规格一致。
